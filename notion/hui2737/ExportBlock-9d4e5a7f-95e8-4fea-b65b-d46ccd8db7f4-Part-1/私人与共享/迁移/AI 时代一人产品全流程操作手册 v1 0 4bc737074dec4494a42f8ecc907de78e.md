# AI 时代一人产品全流程操作手册 v1.0

<aside>
🎯

**适用对象**：编程基础弱、核心能力是 Prompt 工程与 AI 编排的一人开发者。

**工具栈基线**：DS TUI + Claude Code CLI（DeepSeek API）主力开发，Cursor + Opus 辅助调试，GitHub 版本管理，Notion AI Agent 做知识与任务管理。

**使用方式**：8 个阶段串成一个闭环。每阶段未达「通过标准」不进入下一阶段。所有模板可直接复制填空。

# 阶段一：问题嗅觉——如何识别值得做的产品机会

**这个阶段解决什么问题**：把「等灵感」改成「系统性地从信号里捞机会」，让产品想法变成可持续供给的流水线，而不是一次性运气。

## 1. 问题信号源清单

**A. 自己的痛点日志（最高优先级，因为你最懂自己的痛）**

- 存哪里：Notion 单独建一个数据库「痛点池」，字段固定为：`日期 | 场景一句话 | 我当时怎么凑合解决的 | 每周出现几次 | 我愿意为解决它付多少钱`。
- 怎么写：只在痛点发生的当下记，不补记不美化。格式锁定为一句话：`我在【场景】想做【任务】，结果被【障碍】卡住，只能【凑合方案】`。
- 写什么：凡是你嘴里冒出「这破玩意儿怎么这么难用 / 居然没有一个工具能……」的瞬间，立刻记。情绪强度是信号强度。

**B. 社交媒体挖掘（找别人的高频痛）**

- 小红书：搜 `怎么 + [任务] + 太麻烦`、`有没有 + [工具] + 推荐`、`[软件名] + 替代`。看评论区里「跪求」「有没有人知道」「我也是」的扎堆程度。
- 即刻：搜话题圈「独立开发」「AI 工具」「效率工具」，看带「吐槽」「求」「踩坑」的动态。
- Twitter/X：搜 `I wish there was an app that`、`why is there no tool for`、`[competitor] alternative`、`[task] is so annoying`。
- Reddit：去 r/SaaS、r/SideProject、r/[你的领域]，搜 `is there a tool` / `recommend` / `frustrated with`。
- 记录方式：把命中的帖子链接 + 一句话痛点摘要，丢进 Notion「痛点池」，标来源。

**C. AI 工具链的缝隙（你最有优势的矿区）**

- 怎么找：列出你每天用的 AI 工具（ChatGPT、Claude、各种 TUI），每次你需要「手动复制粘贴、手动拼 Prompt、手动导出结果」的环节，就是一个缝隙。
- 三类高价值缝隙：①「最后一公里」——AI 给了结果但要人手动整理成可用格式；②「Prompt 复用」——同一类需求每次都要重新拼 Prompt；③「多工具串联」——结果要在 A 工具产出再喂给 B 工具。

**D. 竞品差评挖掘（现成的需求清单）**

- 去哪看：App Store / Google Play 的 1-3 星评论、Product Hunt 评论区、小红书「XX 软件难用」笔记、G2/Capterra（海外 SaaS）。
- 看什么：只看「我本来想用它做 X，结果它不能 / 太贵 / 太复杂」这类，这就是你的切入点。
- 怎么提炼：把 20-50 条差评全文丢给 AI，让它聚类成「抱怨主题 + 出现频次」。

## 2. 问题质量评估框架（5 个可观察条件）

| 条件 | 可量化判断 | 不合格信号 |
| --- | --- | --- |
| 频率 | 目标人群每周遇到 ≥3 次 | 一年才遇到一次 |
| 痛度 | 用户已在用「凑合方案」（Excel/手动/付费替代） | 没人为它动过手 |
| 付费意愿 | 能找到 ≥3 个已付费的同类证据 | 全网都期待免费 |
| 可被一人做 | 你判断 72h 能做出可用版本 | 需要大量数据/牌照/线下 |
| 触达成本 | 目标人群聚集在你能发声的渠道 | 人群分散无法低成本触达 |

规则：5 条里 < 4 条达标，直接丢弃，不要恋战。

## 3. 「问题」→「产品假设」转化模板

> 我打赌【目标人群】愿意用【产品形态】来解决【高频痛点】，因为现有方案【主要缺陷】，而我能用【你的差异化】做得更好。
> 

## 4. 每周问题挖掘最小流程（≤30 分钟）

- 0-5 分钟：翻本周「痛点池」新增条目。
- 5-20 分钟：按 B 的搜索词，每个渠道扫 5 分钟，命中即存。
- 20-30 分钟：用第 2 节框架给本周 Top 3 问题打分，留 1 个进「候选池」。

## 5. AI 协作

- 批量分析痛点帖：把 20-50 条帖子/评论贴进 DS TUI，Prompt：`你是用户研究分析师。下面是用户评论原文。请：1) 聚类成不超过 8 个抱怨主题；2) 每个主题标注出现次数和一条最典型原话；3) 按出现次数排序。禁止脑补未出现的内容。`
- 提炼高频关键词：`从以上文本提取用户反复使用的名词和动词短语，按频次降序列出前 20 个，标注频次。`
- Notion Agent：让它定期把「痛点池」里本周新增条目按场景聚合成一句话摘要。

<aside>
✅

**阶段一通过标准**：①「痛点池」≥20 条真实记录；② 至少 1 个问题在 5 条评估条件中达标 ≥4 条；③ 已写出该问题的一句话产品假设。

</aside>

# 阶段二：竞品快速侦察——30 分钟摸清一个赛道

**这个阶段解决什么问题**：用 30 分钟而不是 100 页报告，判断「这个方向能不能做、值不值得做」，避免一头扎进死胡同或血海。

## 1. 竞品发现渠道清单

**中国**：①「七麦数据 / 点点数据」搜关键词看同类 App 排名与下载量；② 小红书直接搜你的功能关键词，看带货笔记里在推什么工具；③ 微信搜一搜 + 即刻搜产品名，看真实用户讨论。

**海外**：① Product Hunt 搜关键词，看同类产品的 upvote 和评论；② Google 搜 `best [category] tools 2026` 看测评聚合文；③ Reddit / Indie Hackers 搜产品名看真实口碑。

## 2. 30 分钟竞品速查模板（5 维 × 各 1 分钟判断法）

| 维度 | 1 分钟怎么判断 |
| --- | --- |
| 解决什么问题 | 看落地页首屏大标题，一句话能不能复述 |
| 怎么收费 | 直接翻 Pricing 页，记最低/最高档价格和免费额度 |
| 用户最多的抱怨 | 看差评/评论区聚类，记 Top 3 抱怨 |
| 技术复杂度 | 判断核心是否=「调用 API + 简单界面」，是→你能做 |
| 差异化空间 | 看抱怨里有没有「能被你用 72h 解决的」缺口 |

填空模板（每个竞品一行存 Notion）：

> 竞品：***_｜定位：**｜价格：**｜Top 抱怨：**｜复杂度：低/中/高｜我的缺口：*___
> 

## 3. 竞品情报 AI 协作

- 总结竞品页面：用 Cursor/CC 把落地页文本贴入，Prompt：`提炼这个产品的：1) 一句话定位；2) 核心功能不超过 5 条；3) 定价档位。只用原文信息。`
- 提取差评关键词：把评论贴入 DS TUI，复用阶段一的聚类 Prompt。

## 4. 「不做」的判断信号（命中任一即放弃）

- 头部产品免费且已做到 90 分，你无明显缺口；
- 价格战已打到接近免费，无付费空间；
- 核心壁垒是数据/牌照/线下网络，一人无法搭；
- 抱怨集中在「价格」而非「功能」——说明需求成立但你也赚不到钱。

## 5. Notion 存储结构

建数据库「竞品情报」，字段：`产品名 | 赛道 | 定位 | 价格 | Top抱怨 | 复杂度(低/中/高) | 我的缺口 | 决策(进入/观望/放弃)`。通过关系字段关联到阶段一「候选池」的对应问题。

<aside>
✅

**阶段二通过标准**：①「竞品情报」≥3 个直接竞品已填完 5 维；② 已写出明确的「我的缺口」一句话；③ 决策列给出「进入」。

</aside>

# 阶段三：产品定义——从模糊想法到清晰 Spec

**这个阶段解决什么问题**：把脑子里的模糊想法压缩成一份 AI 能直接照着写代码的 Spec，跳过 PRD 和原型图。

## 1. 一句话产品定义公式

> 【产品名】帮【目标人群】在【场景】下【完成什么】，比【现有方案】更【关键优势】。
> 

**好例子**：

- 「PromptDeck 帮独立开发者把常用 Prompt 一键调用，比每次手敲快 10 倍。」
- 「差评雷达帮 SaaS 开发者自动聚类竞品差评，比手动翻评论省 2 小时。」
- 「周报机帮上班族把碎片记录自动生成周报，比从零写省 30 分钟。」

**坏例子（及为什么坏）**：

- 「一个 AI 驱动的效率平台」——无人群、无场景、无对比。
- 「让生活更美好的工具」——不可验证的空话。
- 「集成多种 AI 能力的超级助手」——边界无限大，一人做不完。

## 2. 用户故事极简写法（≤5 条，格式固定）

> 作为【角色】，我想【动作】，以便【收益】。
> 

例：作为独立开发者，我想一键搜出常用 Prompt，以便不用每次重写。

## 3. MVP 功能边界划定法

规则：只保留「不做它产品就不成立」的功能，其余全进「不做清单」。

**不做清单模板**：

> 本期不做：①登录/账号体系（先用本地存储）；②支付（先验证有没有人用）；③多语言；④数据导出；⑤设置页；⑥【待用户补充：你的产品里诱人但非核心的功能】。理由：每一项都不影响验证「核心假设」是否成立。
> 

## 4. Spec 文档结构（可复制模板）

```
# [产品名] Spec v0.1

## 一句话定义
（套用第1节公式）

## 用户故事（≤5条）
1. 作为___，我想___，以便___
2. ...

## 核心流程（≤7步用户操作）
1. 用户打开页面看到___
2. 用户输入/点击___
3. 系统返回___
... （超过7步说明 MVP 太大，砍）

## 技术选型
- 前端：___
- 后端：___（无则写「纯前端」）
- AI/第三方 API：___
- 部署：___

## 不做清单
- ___（套用第3节模板）

## 完成标准 DoD（每条可验证）
- [ ] 用户能完成核心流程第1-N步不报错
- [ ] 核心功能在___秒内返回结果
- [ ] 移动端浏览器能正常打开
- [ ] 部署到公网，有可访问 URL

## 部署方案
- 平台：___（如 Vercel/Cloudflare Pages）
- 域名：___（先用平台默认子域名）
```

## 5. AI 协作节点（从一句话生成 Spec 初稿）

DS TUI / CC Prompt：`你是产品 Spec 专家。我的一句话想法是：【贴公式结果】。请按我给的模板生成 Spec 初稿。约束：核心流程≤7步；技术选型只用「调用 API + 静态前端」级别的最简方案；主动把你认为非核心的功能放进不做清单。不确定处标注[需我决策]。` 你只做删减和拍板，不要从零写。

## 6. Spec 质量自检清单（答「否」即改）

1. 一句话定义里有没有明确的人群和对比对象？
2. 核心流程是不是 ≤7 步？
3. 每条 DoD 是不是都能客观验证（不是「体验好」这种）？
4. 不做清单里是不是至少砍掉了 3 个诱人功能？
5. 技术选型是不是你判断 AI 能在 72h 内写出来？

<aside>
✅

**阶段三通过标准**：① Spec 六个字段全部填完；② 6 条自检全部为「是」；③ 核心流程 ≤7 步。

</aside>

# 阶段四：技术选型——一人 AI 开发者的技术决策框架

**这个阶段解决什么问题**：不学技术，而是用一套决策树快速锁定「AI 最容易写对、你最容易跑起来」的技术方案。

## 1. 技术选型决策树

- **纯前端工具（无需保存用户数据）**→ 单个 HTML + 原生 JS + Tailwind CDN。为什么：AI 写单文件最稳，无构建步骤，直接拖到 Cloudflare Pages 就上线。复杂界面才上 React + Vite。
- **需要后端的 SaaS**→ Next.js（前后端一体）部署到 Vercel。为什么：一个框架搞定前后端，AI 对它训练充分，Vercel 免费额度够冷启动。
- **需要 AI API 的产品**→ 架构固定为「前端 → 你的后端中转（藏 API Key）→ 模型 API」。**绝不把 API Key 放前端**。后端用 Next.js API Route 或 Cloudflare Workers。
- **特殊形态**：Chrome 扩展（Manifest V3 + 原生 JS，AI 写得好但调试麻烦，需真机测）；CLI 工具（你最熟，Python + Typer / uv，对你最低风险）；Telegram Bot（python-telegram-bot，部署到免费的 serverless）。

## 2. AI 编码难度评级

| 类型 | AI 写得好不好 | 备注 |
| --- | --- | --- |
| 单文件前端 / 纯 UI | ⭐⭐⭐⭐⭐ 极好 | 首选 |
| 调用 REST/AI API | ⭐⭐⭐⭐ 好 | 注意错误处理 |
| CRUD + SQLite | ⭐⭐⭐⭐ 好 | 结构清晰 |
| 鉴权/支付/OAuth | ⭐⭐ 易错 | 用现成服务，别自己写 |
| 复杂状态管理/并发 | ⭐⭐ 易错 | 拆细任务，多审查 |
| 浏览器扩展跨域/权限 | ⭐⭐ 易错 | 真机反复测 |

## 3. 数据库选型极简指南

- **不需要数据库**：数据只在用户本地、刷新可丢→用 localStorage。
- **用 SQLite**：单机/CLI 工具、数据量小、无需多人同时写。
- **上 Supabase**：需要用户账号、跨设备同步、多人访问。免费额度够冷启动，自带认证省掉自己写登录。

## 4. 第三方服务选型原则（只推免费/极低成本）

- 认证：Supabase Auth / Clerk 免费档。别自己写密码哈希。
- 支付：国内用「微信/支付宝个人收款码 + 手动开通」验证期足够；规模化再上「Stripe（海外）」或「面包多/Lemon Squeezy」。
- 邮件：Resend 免费档。
- 部署：Cloudflare Pages / Vercel / Workers 免费档。
- 原则：能用别人现成服务的，绝不自己写；先验证需求，再为付费功能付钱。

## 5. AI 协作

DS TUI Prompt：`我要做【产品一句话定义】，需要保存用户数据吗：【是/否】，目标用户量级：【冷启动几十人】。请按「一人开发者、零预算、AI 辅助编码、72h 出 MVP」约束，给我 2 个技术方案对比（前端/后端/数据库/部署/第三方服务），列出各自的 AI 编码风险点，并推荐 1 个，说明理由。`

<aside>
✅

**阶段四通过标准**：① 技术选型每一栏（前端/后端/AI API/数据库/部署）都有明确选择；② 已确认所选方案全部在免费额度内；③ API Key 中转架构已确定（如涉及）。

</aside>

# 阶段五：开发执行——AI 协作编码的质量控制系统

**这个阶段解决什么问题**：你不懂代码，但要让 AI 写出能跑、少 bug、不跑偏的代码——靠的是流程纪律，不是技术能力。

## 1. Prompt 驱动开发 5 条铁律（正例/反例）

**铁律1：需求 Prompt 要带「上下文 + 约束 + 验收」三件套**

- 正：`基于现有 index.html（已附），新增一个搜索框，输入关键词后过滤下方列表。约束：不引入新依赖，不改动现有样式。完成后告诉我怎么验证。`
- 反：`帮我加个搜索功能` ← 无上下文、无约束，AI 会自由发挥。

**铁律2：任务粒度=「一个可独立验证的功能点」**

- 正：`实现「点击导出按钮，把列表存成 CSV 下载」这一个功能。`
- 反：`把整个导出模块做了` ← 太大，AI 会牵连改动；也别细到 `把这行变量改个名` ← 太碎低效。

**铁律3：代码审查用「让 AI 自证 + 你跑流程」**

- 你不懂代码，就让另一个 AI 审：把改动贴给 Cursor+Opus，Prompt：`审查这段代码：1) 有没有逻辑 bug；2) 有没有未处理的报错；3) 有没有改到不该改的地方。只报问题，不要重写。`
- 然后你亲自按 DoD 跑一遍核心流程，能跑通=过。

**铁律4：处理「自信型错误」——要求 AI 给验证步骤，自己实测**

- 正：每次改完追加 `给我一条可以立即运行的验证命令或操作步骤，证明它真的работает。` 然后你真的去跑。
- 反：看 AI 说「已完成，功能正常」就信 ← 这是翻车头号原因。

**铁律5：防止擅自重构——显式锁定改动范围**

- 正：`只修改 【文件名】 的 【函数名】，其他文件和函数一律不动。如果你认为需要改别处，先告诉我原因，等我确认。`
- 反：`优化一下这个项目` ← AI 会大面积重写，你 review 不过来。

## 2. 工具分工矩阵

| 任务 | 用谁 | 为什么 |
| --- | --- | --- |
| 从 Spec 生成项目骨架、批量写新功能 | Claude Code CLI（DeepSeek） | 多文件、长流程、性价比高 |
| 快速问答、技术选型、方案对比 | DS TUI | 轻量、对话式 |
| 单点调试、定位疑难 bug、代码审查 | Cursor + Opus | 上下文强、主动揪 bug |
| 跨 AI 交叉验证 | CC 写 + Cursor 审 | 互相挑错 |

## 3. Git 极简工作流（5 个命令）

```
git add .                          # 暂存改动
git commit -m "feat: 加搜索功能"   # 提交并写清做了什么
git push                           # 推到 GitHub
git log --oneline                  # 看历史，出事好回退
git checkout .                     # 放弃未提交改动，回到上个 commit
```

提交规范：`feat:` 新功能 / `fix:` 修 bug / `docs:` 文档。**每完成一个可验证功能就 commit 一次**，这是你不懂代码时的「后悔药」。

## 4. 代码质量最低底线

- 核心流程能跑通且不报错；
- API Key 不在前端、不进 Git（用 .env + .gitignore）；
- 用户输入有基本校验（空值/超长）；
- 报错有兜底提示，不是白屏；
- 满足这 4 条即可上线，不追求架构完美。

## 5. 常见翻车场景 × 处理

| 场景 | 处理 |
| --- | --- |
| AI 说「已修复」但还是报错 | 把完整报错原文贴回去，Prompt：`这是完整报错，定位根因，最小修复，不要重构` |
| 改 A 功能改坏了 B 功能 | `git checkout .` 回退，重新用铁律5锁定范围 |
| AI 写了空文件/半截文件 | 写完逐个核对文件非空（你 README 里踩过的坑） |
| API Key 泄露进 Git | 立刻在平台吊销重置 Key，补 .gitignore |
| AI 反复在同一个 bug 上打转 | 换工具（CC↔Cursor），或要求它先只读分析不改代码 |
| 依赖装不上/环境乱 | 回到 uv 标准流程：独立项目目录 + 自己的 pyproject.toml + pin 3.12 |

## 6. 交叉验证法

固定流程：CC 写完一个功能 → 把 diff 贴给 Cursor+Opus 审 → 把 Cursor 提的问题再贴回 CC 修 → 你亲自跑 DoD。一个 AI 当作者，一个 AI 当审稿人，你当终审。

<aside>
✅

**阶段五通过标准**：① 核心流程能在公网 URL 上完整跑通不报错；② API Key 不在前端且不在 Git 历史；③ 每个功能都有对应 commit。

</aside>

# 阶段六：产品打磨——从「能用」到「好用」的最小动作

**这个阶段解决什么问题**：用一人能负担的 80/20 打磨，把「能跑」拉到「别人愿意用」，不做大公司式 UX 工程。

## 1. 上线前必检清单（≤10 条，各 ≤1 分钟）

- [ ]  首屏 3 秒内能看懂「这是什么、给谁用」
- [ ]  核心功能 ≤3 步走通
- [ ]  报错信息是人话（不是英文堆栈）
- [ ]  手机浏览器打开布局不乱
- [ ]  首屏加载 ≤3 秒
- [ ]  有一句话说明「怎么用」（占位提示或示例）
- [ ]  空状态有引导（没数据时不是空白页）
- [ ]  关键按钮点了有反馈（loading/成功提示）
- [ ]  有联系方式/反馈入口（一个邮箱或表单即可）
- [ ]  换个浏览器再测一遍核心流程

## 2. 找人试用执行方案

- 去哪找：自己的微信好友里 3-5 个目标人群、即刻/小红书私信 5 个相关用户、独立开发者社群发「求 5 个人试用」。
- 怎么找：给一句话价值 + 直接给链接，不要让对方注册。
- 问什么（固定 3 问）：①「你第一眼觉得这是干嘛的？」②「你刚才哪一步卡住或犹豫了？」③「你会不会再用第二次，为什么？」
- 怎么记录：Notion 建「试用反馈」表，字段 `用户 | 第一印象 | 卡点 | 会不会再用 | 原话`。

## 3. 反馈分类处理框架

| 反馈类型 | 处理 |
| --- | --- |
| 多人卡在同一步 | 立即改（这是核心可用性问题） |
| 「我想要个 X 功能」单人提 | 记录不改（一个人的需求不代表方向） |
| 「我没看懂这是干嘛的」 | 立即改文案/首屏（定位问题） |
| 「用不上 / 不是我的痛点」多人说 | 警报：可能方向错了，回阶段一 |
| 主观审美吐槽 | 排到最后 |

## 4. UI/UX 最小有效提升（AI 生成，5 个通用动作）

1. 让 AI 套一个干净的设计系统（统一字号、间距、主色），Prompt：`用 Tailwind 给这个页面套一套极简清爽风格：统一留白、一个主色调、按钮和卡片圆角统一。不改功能。`
2. 加 loading 和成功/失败 toast。
3. 首屏加一句话 slogan + 一个示例输入。
4. 空状态加引导文案 + 示例按钮。
5. 移动端单列堆叠，按钮够大可点。

## 5. 性能优化 3 个最低成本动作

- 图片压缩 + 懒加载；
- 用 CDN（Cloudflare Pages 自带）；
- AI 调用加「请求中」状态并防重复点击。

## 6. AI 协作做可用性审查

把首屏截图 + 核心页截图丢给 Cursor/多模态 AI，Prompt：`你是可用性专家。看这张产品截图：1) 用户能不能 3 秒看懂这是什么？2) 有哪些让人犹豫或困惑的地方？3) 给 5 条不需要重做、改文案/布局就能提升的建议。按影响大小排序。`

<aside>
✅

**阶段六通过标准**：① 10 条必检全部通过；② ≥3 个真实用户试用且反馈已记录；③「多人共同卡点」已全部修复。

</aside>

# 阶段七：发布与冷启动——从 0 到前 100 个用户

**这个阶段解决什么问题**：把做好的产品「推出门」，用结构化动作拿到前 100 个用户，而不是发一条没人看的帖子就结束。

## 1. 发布渠道优先级矩阵

| 产品类型  市场 | 中国优先级 | 海外优先级 |
| --- | --- | --- |
| 效率/AI 工具 | 小红书 > 即刻 > V2EX | Twitter > Product Hunt > Reddit |
| 开发者工具 | V2EX > 即刻 > 少数派 | Hacker News > Indie Hackers > PH |
| 内容/泛大众 | 小红书 > 微信群 > 即刻 | Reddit > Twitter > PH |

规则：冷启动只主攻 1 个主渠道 + 1 个备用，别全渠道铺，一人顾不过来。

## 2. 各渠道发布模板（可填空）

**小红书**：

> 标题：`我做了个工具，专治【痛点】（亲测省X分钟）`
> 

> 正文：①我也被【痛点】烦了很久；②试了【现有方案】都不行；③于是自己做了【产品】；④它能【核心功能一句话】；⑤免费用，链接在评论/简介。
> 

> 配图：3-9 张，首图=效果对比或界面截图，要清晰大字。
> 

**即刻/V2EX**：

> 标题：`[作品分享] 我做了个【一句话定义】`
> 

> 正文：背景痛点 → 我做了什么 → 怎么用（带链接）→ 求反馈。
> 

**Product Hunt**：

> Tagline（≤60字符）：`[Product] - [what it does in one line]`
> 

> Description：问题 → 方案 → 关键功能 3 条 → 为谁做。
> 

> 配图：1 张 logo + 3-5 张 GIF/截图。
> 

## 3. Product Hunt 发布 checklist（做海外才需要）

- [ ]  提前 1-2 周注册账号并日常互动养号
- [ ]  准备 logo、首图 gallery、tagline、description、demo 链接
- [ ]  选发布日：周二至周四，太平洋时间 00:01 上线
- [ ]  准备 first comment（创始人故事）
- [ ]  发布前通知你认识的人当天来支持（不刷票，真实互动）
- [ ]  当天全程在评论区回复

## 4. 发布日时间表（以主渠道为小红书为例，北京时间）

- 前一晚：素材、文案、链接全部就位，自己再走一遍核心流程。
- 09:00 发布主渠道帖。
- 09:00-12:00 每 30 分钟回一次评论。
- 12:00 同步发备用渠道。
- 全天：每条评论都回，置顶引导链接。
- 22:00 复盘当天数据。

## 5. 发布后 72 小时监控清单

- 看什么：访问量(UV)、核心动作完成数、注册/留资数、评论与私信。
- 频率：前 24h 每 2-3 小时看一次，之后每天 2 次。
- 立即响应信号：①有人报 bug→立刻修；②评论区同一个问题被反复问→置顶解答；③转化漏斗某一步骤大量流失→当天优化那一步。

## 6. 冷启动失败处理

- **发了没人看**：不是产品问题，是渠道/标题问题。换标题角度（从「我做了X」改成「怎么解决X痛点」），换渠道再发，或找 KOC 帮发。
- **有人看没人用**：首屏没讲清价值或注册门槛太高。砍掉注册、改首屏文案、加示例。
- **有人用没人付费**：先确认免费用户量够不够（太少则继续拉新）；够了还不付费→付费点不痛或定价错，回阶段八调定价。

<aside>
✅

**阶段七通过标准**：① 至少 1 个主渠道完成正式发布；② 拿到首批真实用户（目标 ≥30，理想 100）；③ 发布后 72h 数据已记录且高频 bug 已修。

</aside>

# 阶段八：迭代与增长——从 MVP 到可持续产品

**这个阶段解决什么问题**：用数据而不是情绪，判断「继续做、加注、还是止损」，并设计让产品自己长大的机制。

## 1. 产品健康度看板（5 指标 + 阈值）

| 指标 | 健康 | 警告 | 危险 |
| --- | --- | --- | --- |
| 周活(WAU) | 持续增长 | 持平 | 连续 2 周下降 |
| 次日留存 | ≥40% | 20-40% | <20% |
| 7 日留存 | ≥20% | 10-20% | <10% |
| 付费转化率 | ≥2% | 0.5-2% | <0.5% |
| 反馈情绪 | 主动夸/推荐 | 中性 | 多人吐槽同一点 |

> 假设前提：以上阈值为「工具类小产品冷启动期」经验区间，非保证值；你的赛道可上下浮动，重点看趋势而非绝对数。
> 

## 2. 迭代优先级决策框架（收到 10 个需求怎么排）

打分：`优先级 = (影响人数 × 痛度) / 开发成本`。

- 影响人数：多少比例用户提到/会用（1-5）；
- 痛度：不解决会不会流失（1-5）；
- 开发成本：你估的工时（1-5，越大越贵）。

分高者先做。被单人提、成本高的，一律推后。

## 3. 「止损」判断标准（命中 ≥2 即考虑放弃）

- 留存持续在危险区且改了 2 版无起色；
- 拉新成本 > 你能承受，且无自然增长；
- 你自己都不想用了 / 没有内在动力；
- 跑了 ≥1 个月核心假设被证伪（没人为它付费/复用）。

## 4. 「加注」判断标准（命中 ≥2 可投入更多）

- 留存进入健康区且自然增长；
- 有用户主动推荐/自来水；
- 付费转化 ≥ 阈值且有人催你加功能；
- 你做起来有心流、不觉得勉强。

## 5. 三种定价模型（适合一人开发者）

| 模型 | 适用场景 | 定价公式 |
| --- | --- | --- |
| Freemium（免费+高级付费） | 有明确「重度用户」需求 | 免费档够轻度用；付费 = 解锁高频刚需功能，定价锚定「省下的时间×时薪」 |
| 一次性买断 | 工具型、无持续服务成本 | 价格 ≈ 用户一个月用它省下的钱 |
| 用量计费/订阅 | 持续调用 AI API、有边际成本 | 定价 = API 成本 × (2~3) + 你的利润 |

> 假设前提：定价需用真实成本和至少 5 个用户的付费意愿访谈校准；上述公式是起点不是终值。
> 

## 6. 增长飞轮（3 个可套用结构）

1. **产出即传播**：用户用你的工具产出的结果自带「由 XX 生成」水印/链接→看到的人来用。
2. **邀请解锁**：邀请 N 个好友解锁高级功能/额度→老用户帮你拉新。
3. **内容飞轮**：你把「用产品解决真实问题」的过程发小红书/Twitter→引流→用户反馈→变成新内容素材。

## 7. AI 协作

- 分析反馈：把「试用反馈」「用户私信」批量贴给 DS TUI，Prompt：`聚类这些反馈成主题，标频次和情绪（正/中/负），并按第2节公式给每个主题打优先级建议。`
- 生成迭代优先级：把需求清单 + 你的工时估计贴入，让 AI 套打分公式排序。
- 自动生成 changelog：把本期 git commit 列表贴给 AI，Prompt：`把这些 commit 整理成给用户看的更新日志，分新功能/优化/修复三类，用人话，不要技术术语。`

✅
