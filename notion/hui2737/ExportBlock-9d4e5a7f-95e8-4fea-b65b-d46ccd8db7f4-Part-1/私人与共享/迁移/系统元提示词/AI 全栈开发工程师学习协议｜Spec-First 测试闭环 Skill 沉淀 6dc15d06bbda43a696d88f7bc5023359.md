# AI 全栈开发工程师学习协议｜Spec-First 测试闭环 Skill 沉淀

🧑‍💻

## 核心结论

这篇文章最值得吸收的不是 TRAE 这个具体工具，而是一个可迁移的 AI 全栈开发范式：**Spec 先行定义边界，测试验证提供反馈，多仓库上下文保证前后端同步，失败 Case 沉淀成 Skill。** 以后学习 AI 全栈开发，不能停留在“会写 Prompt 让 AI 生成代码”，而要训练自己成为能设计需求规格、验收标准、测试闭环和复用资产的工程负责人。

## 最佳实践 1：从 Vibe Coding 转向 Spec-First

Vibe Coding 适合探索原型，但项目复杂后会出现三个问题：AI 生成代码无法自动验证、复杂功能反复返工、前后端接口手动对齐。Spec-First 的价值是把“模糊需求”变成 AI 可以执行的工程契约。

| 文档 | 作用 | 必须包含 |
| --- | --- | --- |
| [spec.md](http://spec.md) | 定义需求和技术边界 | 功能目标、接口定义、数据结构、前后端分工、约束、非目标 |
| [tasks.md](http://tasks.md) | 拆成可执行任务 | 优先级、依赖顺序、原子任务、预计改动范围 |
| [checklist.md](http://checklist.md) | 定义验收标准 | 完成标准、测试样例、回归项、失败条件 |

**学习动作：**以后每做一个 AI Coding 需求，先写三件套，不允许直接让 AI 改代码。完成标准是：另一个 AI 或开发者只看 [spec.md](http://spec.md)、[tasks.md](http://tasks.md)、[checklist.md](http://checklist.md)，也能理解要做什么、怎么做、做到什么算完成。

## 最佳实践 2：全栈上下文必须前后端同屏

多仓库开发的关键不是“打开多个仓库”，而是让 AI 同时看到产品需求、前端页面、后端接口、数据结构、调用链和工作区规则。否则前端改了字段，后端没跟；后端改了接口，前端没同步；最后人被迫做接口胶水。

| 场景 | 正确做法 | 风险 |
| --- | --- | --- |
| 前后端联调 | 同一工作区放前端、后端、接口文档 | Spec 可能只生成在一个仓库，需要手动指定同步位置 |
| 代码仓库很大 | 沉淀项目知识库和路径索引 | AI 找错文件、误判依赖 |
| 复杂需求 | 一次 Prompt 先生成全链路方案，再分任务执行 | 上下文过大导致遗漏，必须用 checklist 回收 |

**学习动作：**建立自己的项目工作区规则：目录结构、启动命令、测试命令、接口约定、禁止改动区域、常见坑。以后每个项目第一步不是写代码，而是让 AI 读懂工作区规则。

## 最佳实践 3：测试不是附加动作，而是 AI Review 的核心

不要问 AI“这段代码对不对”，要给它输入、期望输出和可执行测试，让它自己运行、失败、修复、再运行。AI Coding 的质量不是靠自信解释，而是靠测试反馈。

### 后端验证

最适合从纯函数、解析逻辑、接口返回处理、数据转换开始。正确格式：

```
输入样例：input1.txt
期望输出：output1_text.txt / output1_images.txt
通过标准：文本正确率 > 95%，图片召回率 > 95%
执行要求：AI 必须自动运行测试，失败后修改代码，直到通过或说明不可通过原因。
```

关键教训：只给输入不给期望输出，AI 会凭感觉实现；补充明确输出后，验证才稳定。

### 前端验证

前端不要只写测试说明，必须写能真实执行的 Playwright 测试：

```
启动后端 → 启动前端 → 打开真实页面 → 操作真实按钮 / 表单 / 上传控件 → 观察真实请求 → 验证页面状态和接口结果。
```

Playwright 脚本适合长期重复验证；MCP / CUA 更适合一次性探索。长期项目优先沉淀可执行脚本。

## 最佳实践 4：失败 Case 要沉淀成 Skill

一次 Prompt 解决的是当前问题；Skill 解决的是下一次同类问题。每次 AI 卡住，都要判断是需求不清、上下文不足、测试不可执行、权限确认阻塞、路径不清，还是验收标准缺失。能重复出现的问题必须沉淀为 Skill、项目规则、测试模板或 checklist。

| 失败类型 | 沉淀方式 |
| --- | --- |
| AI 不知道改哪里 | 补项目路径索引 / 代码地图 |
| AI 写了但没跑 | 补“必须执行测试并贴结果”的 Skill 规则 |
| 前后端字段不一致 | 补接口契约和联调 checklist |
| 测试依赖人工操作 | 改成 Playwright 或脚本化验证 |
| 重复创建测试数据 | 沉淀测试数据生成 Skill |

## 适用边界

这套工作流在新项目、MVP、独立模块、逻辑清晰、接口边界明确的小团队场景收益最高。对存量业务、历史包袱重、隐性规则多、强合规、核心生产链路，不能直接放大自动化；要先做沙盒、最小切片、只读分析和小范围测试。

**判断公式：**

```
AI Coding ROI = 需求清晰度 × 测试可执行性 × 上下文完整度 × 失败可回滚性 ÷ 历史包袱复杂度
```

如果没有测试、没有 Spec、没有回滚，就不要把 AI 生成代码当成完成。

## AI 全栈开发工程师学习路线

### 第 1 阶段：Spec 能力

目标：能把任意功能需求写成工程契约。

- 练习：每周选 3 个小功能，写 [spec.md](http://spec.md)、[tasks.md](http://tasks.md)、[checklist.md](http://checklist.md)。
- 完成标准：AI 能按文档实现，不需要你反复补充“不是这个意思”。
- 重点能力：需求边界、接口定义、非目标、验收标准。

### 第 2 阶段：后端可验证实现

目标：让 AI 写后端代码时必须有输入输出样例和自动测试。

- 练习：做 5 个纯函数 / 解析 / 数据转换任务。
- 完成标准：每个任务都有样例输入、期望输出、测试脚本和运行结果。
- 重点能力：测试数据设计、边界 Case、回归验证。

### 第 3 阶段：前端真实浏览器测试

目标：用 Playwright 验证真实用户路径，而不是只看页面“像不像”。

- 练习：写 3 条 E2E 测试：登录 / 表单提交 / 上传或列表操作。
- 完成标准：AI 能启动服务、运行测试、根据失败日志修复。
- 重点能力：选择器稳定性、等待策略、接口观测、状态断言。

### 第 4 阶段：全栈多仓库闭环

目标：一次需求能同时改前端、后端、接口和测试。

- 练习：做 1 个完整小功能：新增字段 / 新增接口 / 页面展示 / E2E 验证。
- 完成标准：Spec、代码、测试、文档同步更新。
- 重点能力：上下文组织、任务拆解、依赖顺序、回滚意识。

### 第 5 阶段：Skill 沉淀

目标：把重复工作变成可复用能力。

- 练习：把“写 Spec”“生成测试样例”“运行 Playwright”“接口联调检查”各沉淀成一个 Skill / 模板。
- 完成标准：下一次同类需求能直接调用，不从零写 Prompt。
- 重点能力：失败 Case 归因、规则抽象、checklist 维护。

## 个人默认开发协议

```
每个 AI Coding 需求必须按以下顺序执行：
1. 写 Spec：需求、接口、数据结构、前后端边界、非目标。
2. 写 Tasks：拆成原子任务，标出依赖顺序。
3. 写 Checklist：定义完成标准、测试样例、回归项。
4. 先补测试：后端给输入输出，前端给 Playwright 场景。
5. 再写代码：每轮只改一个明确任务。
6. 必须运行：AI 要执行测试并返回结果，不只解释。
7. 失败沉淀：失败 Case 进入测试、规则或 Skill。
8. 完成定义：代码通过测试 + 文档同步 + 无关键回归。
```

## 下一次实战任务

用一个最小全栈功能训练这套方法：选择“登录页表单校验 / 文件上传 / 列表筛选 / 新增字段展示”四选一。要求不是做大项目，而是完整走完 Spec → Test → Code → Run → Fix → Skill，一次形成可复用模板。
