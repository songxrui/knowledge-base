# 人生OS v2.0

<aside>
🧠

**Cognitive OS v2.0｜统一系统说明书**

- 系统内核：**问题 → 模型 → 实验 → 复利** 的认知编译器。
- 主路径：**Inbox → Questions → Models → Experiments → 回写 Models → 输出内容**。
- 思想底层：**Munger**（多元模型 + 反转） × **Naval**（杠杆 + 复利） × **Dan Koe**（创作 = 学习 = 变现）。
- 关键结论标注证据强度（E0–E3），并给出反对观点与失效条件。
- 所有建议必须可执行可验证（DoD）。
- 优先最小复杂度，避免数据库膨胀。
</aside>

## 当前阶段唯一主线（吸收自健康索引）

### 核心排序

1. **唯一主线**：恢复并长期维持健康高水平状态
2. **证据层**：记录、回测、整理真实过程
3. **沉淀层**：把有效策略写成模型、规则和专题页
4. **下游层**：未来内容、产品、市场化

### 当前阶段判断标准

- 它是否真的让我更接近健康巅峰？
- 它是否帮助我长期维持那个状态？
- 它是在服务主线，还是在偷换成内容 / 产品 / 市场游戏？

## 运行约束（吸收自外置大脑）

- 不确定信息必须标注，不得猜测
- 对关键结论标注证据强度（E0–E3）
- 提供反对观点与失效条件
- 不假设缺失背景信息
- 所有建议必须可执行可验证（给出 DoD）
- 优先最小复杂度方案
- 指出潜在认知偏差与风险

### 系统哲学（三条公理）

1. **未经验证的知识 = 负债**（Munger）→ Model 没有 Experiment 链接 = 负债，不是资产。
2. **复利只发生在可复用的抽象层**（Naval）→ Models 库是唯一复利层；其他库都是管道。
3. **输出倒逼输入质量**（Dan Koe）→ 每周 ≥1 次公开输出，强制压缩与校准。

### 证据强度（用于关键结论标注）

- **E0**：结构设计选择（不依赖个人背景也成立）
- **E1**：来自你明确提供的偏好/约束，但可能变化
- **E2**：多处信息一致指向的稳定特征（仍可能变化）
- **E3**：可由系统内记录直接验证（计数/状态/周期）

## 1️⃣ 整体架构（四库 + 输出层）

### 架构图

```
📥 Inbox ─── 缓冲 + 证据（已合并 Assets）
   │
   ▼
❓ Questions ─── 问题驱动 + 优先级中心
   │         ╲
   ▼          ▼
🧩 Models ←── 🧪 Experiments
   │    ╲         │
   ▼     ▼        ▼
📤 OUTPUT（内容卡片 / 即刻 / 小红书）
   │
   ▼
🔄 外部反馈 → 新 Inbox 条目 → 循环
```

### 模块职责

- [Inbox（输入箱）](Inbox%EF%BC%88%E8%BE%93%E5%85%A5%E7%AE%B1%EF%BC%89%2014903c54805d833da67e813cfc1d6865.md)：碎片输入的缓冲池 **+ 证据素材库**（已合并原 Assets 功能）。
- [Questions（问题库）](Questions%EF%BC%88%E9%97%AE%E9%A2%98%E5%BA%93%EF%BC%89%20f8a03c54805d83308a65811defa1dd4e.md)：问题入口与优先级中心，负责聚焦与决策。
- [Models（模型库）](Models%EF%BC%88%E6%A8%A1%E5%9E%8B%E5%BA%93%EF%BC%89%2093803c54805d83cba700810cd1f905d2.md)：可复用模型（**复利核心**），负责抽象沉淀。
- [Experiments（实践/实验）](Experiments%EF%BC%88%E5%AE%9E%E8%B7%B5%20%E5%AE%9E%E9%AA%8C%EF%BC%89%20d4e03c54805d8342b0a881dfcc155ce0.md)：实践验证层，负责产出可验证结果并回写模型。

> **关键结论**：系统资产不是"素材"，而是"被实践验证过、可复用的模型"。**证据强度：E0**
> 

### 认知目标函数

$$
U = \frac{V \times R \times C}{F \times D}
$$

| 变量 | 含义 | 度量方式 |
| --- | --- | --- |
| **V** Verified Models | 被实验验证过的模型数 | Models 中 Experiment ≥ 1 的条目 |
| **R** Reuse Count | 模型被不同 Question 复用的次数 | Models.Linked Questions 平均链接数 |
| **C** Compression | 信息压缩效率 | 本月 Done 的 Inbox / 新增 Model 数 |
| **F** Friction | 执行摩擦 | Inbox New > 7天 条目数 + Experiments 无 Review Date 数 |
| **D** Decay | 系统退化 | Models 中 > 90天未被 Experiment 引用的僵尸模型数 |

> 每周复盘只看这 5 个数。数字健康 = 系统健康。

## 2️⃣ 信息流（五阶段闭环）

```
Stage 1: CAPTURE（捕获）
   输入 → Inbox（≤ 1 句话 + Type + Reliability）
   硬规则：7 天未处理 → 删除/Done
        │
        ▼
Stage 2: COMPRESS（压缩）
   Inbox → Question（改写为疑问句 + Success Metric）
   Munger 检查：写 Anti-Thesis
        │
        ▼
Stage 3: CONNECT（连接）
   Question → 关联 ≥1 个 Model
   没有匹配 → 创建草稿 Model（标题前缀"草稿："）
   Naval 检查：Leverage Type 是什么？
        │
        ▼
Stage 4: TEST（验证）
   Model → Experiment（Hypothesis + Protocol + Metric + Review Date）
   硬规则：无 Metric 不 Doing；≤14 天必须出 Result
        │
        ▼
Stage 5: COMPOUND（复利）
   5a. 内部复利：Result → 回写 Model（Steps / Failure Modes / Last Verified）
   5b. 外部复利：Verified Model → 输出内容卡片（Dan Koe 闭环）
        │
        ▼
   外部反馈 → 新 Inbox 条目 → 回到 Stage 1
```

> **关键结论**：没有 Experiment 的问题，长期会退化为"信息坟墓式思考"。没有输出的模型，缺乏外部校准，会导致认知近亲繁殖。**证据强度：E0**
>

## 3️⃣ 极简数据库设计

### DB1：Inbox（输入箱 + 证据层）

**用途**：收集碎片输入 **并管理证据素材**，快速分流到问题/实验/模型。

**字段**

| 字段名 | 类型 | 作用 |
| --- | --- | --- |
| `Title` | title | 输入的一句话标题 |
| `Type` | select（Idea / Info / Task / Quote / Observation / Article / Data） | 输入类别。前 5 种 = 碎片输入；Article / Data = 证据素材 |
| `Source` | url | 来源链接（可空） |
| `Status` | status（New / Processing / Done） | 处理状态 |
| `Reliability` | select（High / Medium / Low / Unknown） | 证据可信度评级 |
| `Key Takeaway` | text | 一句话结论（压缩训练） |
| `Linked Question` | relation → Questions | 这条输入最终归属的问题 |

**与其他库关联**：Inbox → Questions 通过 `Linked Question`。

**DoD**

- 每条 Inbox 最迟 7 天内必须：链接到一个 Question 或标记为 Done。
- `Reliability = Unknown` 的证据不允许直接支持 P0 结论。

**反对观点与失效条件**

- 反对：直接把输入写进问题库即可。
- 失效：如果不做分流，Inbox 会膨胀为"收藏夹"。

---

### DB2：Questions（问题库）

**用途**：以问题驱动认知与决策，把注意力聚焦到少数关键问题。

**字段**

| 字段名 | 类型 | 作用 |
| --- | --- | --- |
| `Question` | title | 问题句式标题（如何/为什么/是否…） |
| `Domain` | select（投资 / 训练 / 认知 / 技能 / 生活系统） | 领域归类 |
| `Priority` | select（Now / Next / Later） | 优先级 |
| `Success Metric` | text | 成功判定（必须可验证） |
| `Anti-Thesis` | text | **反对观点**（P0 必填，Munger 反转原则） |
| `Output Status` | select（Not Yet / Drafted / Published） | 是否已转化为公开内容（Dan Koe 闭环） |
| `Related Models` | relation → Models | 该问题使用/复用的模型 |

**DoD**

- `Priority=Now` 的问题必须写 `Success Metric` + `Anti-Thesis`，且至少关联 1 个 Experiment。
- Question 被 Close 时，Output Status = Not Yet → 提示"是否输出为内容？"

**反对观点与失效条件**

- 反对：用待办清单更直接。
- 失效：如果 `Success Metric` 写成模糊口号，系统会变成"自我安慰"。

---

### DB3：Models（模型库）

**用途**：沉淀可复用的思考与行动模板，实现跨领域迁移与复利。

**字段**

| 字段名 | 类型 | 作用 |
| --- | --- | --- |
| `Model Name` | title | 模型名 |
| `Model Type` | select（Decision / Learning / Training / Investment / Meta） | 模型类型 |
| `Leverage Type` | select（Labor / Capital / Code / Media） | Naval 杠杆分类 |
| `When to Use` | text | 适用场景（≥2 个不同 Domain） |
| `Anti-Use` | text | **不适用场景**（Munger 能力圈边界） |
| `Steps` | text | 编号步骤（1/2/3…） |
| `Failure Modes` | text | 失效条件与风险 |
| `Last Verified` | date | 上次被 Experiment 验证的日期 |
| `Linked Questions` | relation → Questions | 该模型服务的问题集合 |

**DoD**

- 每个模型至少包含：When to Use + Anti-Use + Steps + Failure Modes。
- `Last Verified` 超过 90 天 → 标记为"待复盘"（僵尸模型预警）。
- `Linked Questions` 为空 → 孤岛模型，30 天后删除或归档。

**反对观点与失效条件**

- 反对：模型会变成空洞方法论。
- 失效：没有被 Experiments 验证、也不回写更新 → 模型僵化。

---

### DB4：Experiments（实践/实验）

**用途**：用最小成本验证模型/假设，产出可复盘的结果。

**字段**

| 字段名 | 类型 | 作用 |
| --- | --- | --- |
| `Experiment` | title | 实验名（动词 + 周期） |
| `Linked Question` | relation → Questions | 对应问题 |
| `Used Models` | relation → Models | 使用的模型 |
| `Hypothesis` | text | 如果做X→Y会变化，因为Z |
| `Protocol` | text | 执行方案（≤3 步） |
| `Metric` | text | 主指标（1 个） |
| `Review Date` | date | 复盘日期 |
| `Result` | select（Positive / Negative / Inconclusive） | 结果 |

**Result → 回写协议**

| Result | 回写动作 |
| --- | --- |
| Positive | 更新 Model.Steps + Last Verified |
| Negative | 更新 Model.Failure Modes + Anti-Use |
| Inconclusive | 拆分为 2 个更小实验 |

**DoD**

- 任何实验必须设置 `Review Date`；周期默认 7 天，超过 14 天必须拆分。
- 到了 Review Date：必须填 `Result`，并回写到相关 Model。

## 4️⃣ 决策引擎（分层决策协议）

| 决策层 | 特征 | 协议 | 时间预算 |
| --- | --- | --- | --- |
| **L0 默认** | 可逆、低风险 | 直接执行，不进 Questions | < 2 min |
| **L1 轻量** | 可逆、中等影响 | 写 Question + 关联已有 Model → 执行 | < 15 min |
| **L2 标准** | 部分不可逆 | Q + M + 7天 Experiment → Result → 决策 | 1 周 |
| **L3 重大** | 不可逆、人生级 | Q + M + Anti-Thesis + 正反双实验 + 24h 冷静期 | 2–4 周 |

> 判定规则（Naval 式）：1 年后无关紧要 → L0/L1；不可逆 → L2/L3。
>

## 5️⃣ 输出引擎（Dan Koe 创作闭环）

<aside>
📤

**核心原则**：每一个被验证的 Model，都必须能压缩为 1 条可发布的内容卡片。写不出 = 还没真正理解。

</aside>

### 内容卡片模板

```
[Hook]  大多数人以为 [旧认知]…
[Break] 但 [你验证过的新认知]
[Model] 核心模型：[Model Name]
[Evidence] 我实验了 [X天]，结果 [Y指标变化了 Z%]
[Action] 你可以这样用：[Steps 简化版]
```

### 输出飞轮

```
Questions(Closed + Result=Positive) → Model(Verified) → 内容卡片 → 发布
→ 外部反馈 → 新 Inbox(Type=Observation) → 回到 Stage 1
```

## 6️⃣ 标签与链接策略

### 最小标签体系

- `A/` 领域：A/投资 A/训练 A/认知 A/技能 A/生活系统
- `M/` 方法：M/实验 M/复盘 M/对比 M/拆解
- `C/` 概念：按需增长（如 C/注意力 C/风险管理）

### 双向链接原则（硬规则）

- Inbox 每条输入 → 最终链接到 1 个 Question。
- `Priority=Now` 的 Question → 有 Success Metric + Anti-Thesis + ≥1 个 Experiment。
- 每个 Model → 至少服务 1 个 Question（`Linked Questions` 非空）。
- 每个 Model → `When to Use` 写 ≥2 个不同 Domain 的应用场景（Munger 格栅）。

### 模型复用机制

- 新问题 → **先在 Models 里找可复用项**，再开 Experiment；不要先写长文总结。

## 7️⃣ AI 可读取结构优化

- **Question 标题**：用"如何/为什么/是否…"开头；避免"XX笔记"。
- **Experiment 标题**：动词开头 + 周期 + Review Date。
- **Steps**：统一编号（1.2.3.），每步一句动词。
- **Failure Modes**：失效条件 + 风险 + 反例（哪怕一句话）。
- **Success Metric / Metric**：可计数或可观察；否则标注为不确定并拆小。

## 8️⃣ 日常运行流程

### 每日（30 分钟）

1. **收集（2 分钟）**：新输入进 Inbox，填 Type + Status=New。证据类填 Reliability + Key Takeaway。
2. **分流（3 分钟）**：从 Inbox 选 1 条 → 链接到一个 Question。
3. **执行（20 分钟）**：只推进 1 个 Now 问题相关的一个动作。
4. **记录（5 分钟）**：在对应 Experiment 补充进展。

**DoD**：每天至少 1 条 Inbox 分流 + 1 次推进 + 1 条可追溯记录。

### 每周（30–45 分钟，固定一天）

1. 打开 Experiments 的 `To Review` 视图，按 Review Date 逐条填 Result。
2. 从本周实验回写 ≥1 个 Model（Steps / Failure Modes / Last Verified）。
3. Questions：只保留 1 个 Priority=Now，其余降级。
4. **输出检查**：本周是否有 ≥1 条公开内容发布？（Output Status 追踪）
5. 查看 5 个健康指标（见下方抗退化机制）。

## 9️⃣ 系统抗退化机制

### 五个健康指标（每周复盘看板）

| 指标 | 🟢 健康 | 🟡 黄灯 | 🔴 红灯 |
| --- | --- | --- | --- |
| Inbox 堆积（New） | ≤ 5 | 6–10 | > 10 |
| P0 数量 | = 1 | = 2 | ≥ 3 |
| 僵尸模型率 | 0 | 1–2 个 | ≥ 3 个（Last Verified > 90天） |
| 实验完成率 | ≥ 80% | 60–80% | < 60% |
| 周输出率 | ≥ 1 条公开内容 | 0 条但有 Draft | 连续 2 周 0 输出 |

### 三层免疫响应

**Layer 1 — 自动清理**

- Inbox New > 7天 → 必须处理
- Model Last Verified > 90天 → 标记"待复盘"
- Experiment 无 Review Date → 不允许创建

**Layer 2 — 周复盘**

- 检查 5 个健康指标
- P0 只保留 1 个
- 回写至少 1 个 Model

**Layer 3 — 月度重构**

- 删除所有孤岛 Model（Linked Questions = 0 且 > 30天）
- 归档所有 Closed > 60天的 Questions
- 复盘 Leverage Type 分布：如果 90% 是 Labor → 系统没在积累杠杆

### 三条硬规则

1. **Inbox**：New 超过 7 天仍未链接 Question → 必须处理。
2. **Experiments**：没有 Review Date 不允许创建；没有 Result 不允许"结束"。
3. **Questions**：Now 永远只有一个。

### 复杂度控制

| 规则 | 阈值 | 违反时 |
| --- | --- | --- |
| 数据库总数 | ≤ 4 | 拒绝新建，先合并 |
| 每库字段数 | ≤ 10 | 新增必须删旧 |
| 标签总数 | ≤ 30 | 合并同义标签 |
| 视图总数/库 | ≤ 5 | 删除最低使用率 |
| 每日系统时间 | ≤ 30 min | 超过说明系统太复杂 |

### 常见认知偏差（提醒 + 对策）

- **确认偏误**：只收集支持观点 → P0 必须填 Anti-Thesis。
- **新奇偏误**：不断开新坑 → 新输入先入 Inbox，不允许直接升级为 Now。
- **计划谬误**：高估执行力 → Review Date 强制复盘，实验周期 ≤14 天。
- **沉没成本**：错误实验不愿放弃 → Result=Negative 时强制 Close 并回写 Failure Modes。

---

## 🔟 长期扩展策略

- **优先扩展"视图/命名规范/模型模板"**，不扩展数据库（E0）。
- 只有当出现新实体类型并稳定存在 ≥8 周，才考虑新增数据库。
- **拒绝扩展条件**：写不出 DoD；Inbox/To Review 堆积未清；新增模块在替代行动痛苦。

## 11️⃣ 吸收自《外置大脑》的补充细则

### 结构兼容说明

- ‣ 中的 `Assets` 独立数据库，在 v2.0 中继续按“**证据层**”保留，但实现上并入 Inbox；规则不变：`Reliability` 必填、`Unknown` 不直接支撑 P0、游离素材 7 天内必须归属或清理。
- Dashboard 继续坚持“只看今天”：只承载链接视图，不承担沉淀与归档。

### 不确定性协议（升级为系统硬约束）

- 不确定信息必须显式标注，不得猜测。
- Inbox 里的证据项允许 `Reliability = Unknown`，但只能作为待验证线索，不能直接支撑关键结论。
- Experiments 里的 `Result = Inconclusive` 合法化“暂无结论”，避免被迫得出伪确定性。
- Models 必须通过 `Anti-Use` / `Failure Modes` 写清边界，避免把局部经验误当通用规律。

> **关键结论**：把“未知、未证实、未收敛”当成一等公民，系统才不会演化成高密度幻觉生成器。**证据强度：E0**
> 

### 标签与领域的最小标准

- `A/` 领域：A/健康、A/职业、A/关系、A/金钱、A/学习、A/生活系统
- `M/` 方法：M/实验、M/复盘、M/对比、M/拆解
- `C/` 概念：按需增长，如 C/注意力、C/习惯、C/力量训练
- 新增标签前先查重；单条证据标签 ≤ 5；标签只做检索与聚类，**真正复用靠 Models**。

### Dashboard（今日）最小布局

1. 今日唯一 Now / P0 问题
2. 正在进行的 Experiments（Doing / To Review）
3. 待复用或 Active 的 Models

### 三个模板原型（保留为最低可用格式）

**Question 原型**

- 标题：用疑问句写（如何 / 为什么 / 是否 / 我该不该…）
- 至少补齐：`Success Metric` + `Anti-Thesis` + 下一步动作

**Model 原型**

- 用途：当我不知道方法是否有效时，用 7–14 天最小实验验证
- 至少补齐：`When to Use`、`Anti-Use`、`Steps`、`Failure Modes`

**Experiment 原型**

- 标题：动词开头 + 周期（如“连续 7 天：每天 10 分钟推进唯一 Now 问题”）
- 至少补齐：`Hypothesis`、`Protocol`、`Metric`、`Review Date`
