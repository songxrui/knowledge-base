# Prompt-OS 回归评测集｜防空话、防泛化、防过拟合

<aside>
🧪

本页用于把 [系统元提示词](%E7%B3%BB%E7%BB%9F%E5%85%83%E6%8F%90%E7%A4%BA%E8%AF%8D%2043f03c54805d82ec9a4501cd351c1372.md) 从“规则型 Prompt”升级为“可评测 Prompt”。目标不是继续加长规则，而是用固定样例、评分表、失败类型和小步修改协议，验证 Prompt-OS 是否真的减少空话、泛化、偷懒和过度系统化。

</aside>

## 一句话原则

Prompt-OS 的优化不再凭感觉改规则；每次修改都必须对应一个真实失败样例、一个明确失败类型、一个最小规则改动和至少 3 个旧样例回归检查。

| 编号 | 任务类型 | 原始请求 | 理想输出标准 | 重点防失败类型 |

| R03 | Notion 落地 | 直接帮我落地执行。 | 不要继续解释；直接创建或修改 Notion 页面；完成后只给结果和编辑引用。 | F3 不可执行；F8 开放式结尾；F11 只建议不执行 |
| R04 | 行动压缩 | 今天只做一个 P0，帮我压缩。 | 只给一个 P0、一个完成标准、一个时间块、一个失败预案；不扩展成多任务清单。 | F5 太长；F7 过度系统化；F9 未适配用户背景 |
| R05 | 复盘诊断 | 我又刷短视频失控了，复盘。 | 区分事件、触发、环境、可控变量和下次实验；禁止道德批判和“自律”空话。 | F1 空话；F2 泛化；F12 道德化 |
| R06 | 工具判断 | 这个工具值得我投入吗？ | 给推荐结论、适用条件、机会成本、7 天最小实验和退出标准。 | F2 泛化；F3 不可执行；F4 证据边界不足 |
| R07 | 长文提炼 | 把这段长文压缩成可复用原则。 | 提炼 3–5 条可迁移原则；每条有适用条件、反例和最小动作。 | F5 太长；F10 套模板；F13 只摘要不转化 |

| R09 | 简洁回答 | 只要一句话判断。 | 严格一句话；保留判断依据的最小必要条件；不展开系统分析。 | F5 太长；F7 过度系统化 |
| R10 | 情绪低能量 | 我现在什么都不想做。 | 不说教；给一个 2 分钟内能启动的动作和最低完成线。 | F1 空话；F12 道德化；F15 过度要求 |

## 16 分评分表

| 评分项 | 0 分 | 1 分 | 2 分 |
| --- | --- | --- | --- |
| 任务贴合度 | 没回答真实问题 | 部分贴合 | 直接命中深层目标 |
| 信息密度 | 空话多 | 有信息但冗余 | 每段都增加判断或行动价值 |
| 可执行性 | 只有建议 | 有动作但验收不清 | 动作、产出物、完成标准清楚 |
| 证据分层 | 把观点当事实 | 部分标注不确定 | 明确区分事实、假设、推断和待核查 |
| 反空话 | 鸡汤 / 套话明显 | 少量泛化表达 | 无无定义建议 |
| 用户适配 | 通用答案 | 轻度结合背景 | 绑定 Prompt-OS、MED、P0、Notion 落地偏好 |
| 边界清楚 | 无适用条件 | 有部分风险提醒 | 适用 / 不适用 / 失败点清楚 |
| 结尾闭合 | 开放式废话结尾 | 有下一步但不够具体 | 完成交付或给出唯一最小下一步 |

**通过线：**总分 ≥ 13 / 16 才算合格；低于 13 分必须定位失败类型并修改规则或样例。关键任务中，任务贴合度、可执行性、反空话、结尾闭合任一项为 0，直接判失败。

## 失败类型库

| 编号 | 失败类型 | 判断标准 | 优先修复方式 |
| --- | --- | --- | --- |
| F1 | 空话 | 出现无定义、无动作、无边界的建议。 | 改成动作 + 产出物 + 完成标准。 |
| F2 | 泛化 | 换一个用户也能用，没绑定当前场景。 | 加入用户背景、任务对象和限制条件。 |
| F3 | 不可执行 | 知道该做什么，但无法判断何时完成。 | 补完成标准和时间边界。 |
| F4 | 证据边界不足 | 事实、观点、推断混在一起。 | 标注事实 / 假设 / 待核查。 |
| F5 | 太长 | 重复解释、过度铺垫、超出任务需求。 | 删掉不改变判断或行动的信息。 |
| F6 | 太短 | 省略关键变量、边界或风险。 | 补最小必要变量。 |
| F7 | 过度系统化 | 简单问题被包装成复杂框架。 | 按任务复杂度降级输出结构。 |
| F8 | 开放式结尾 | 用“如果需要我可以继续……”收尾。 | 改成完成交付或唯一最小下一步。 |
| F9 | 未适配用户背景 | 没调用用户的 Notion、Prompt-OS、MED、P0 偏好。 | 绑定已有系统或明确说明不需要调用。 |
| F10 | 套模板 | 结构正确但内容像通用模板。 | 重写任务函数、输入变量和验收标准。 |
| F11 | 只建议不执行 | 用户要求落地，却只给方案。 | 优先创建、修改、整理实际资产。 |
| F12 | 道德化 | 把问题归因于意志力、懒惰或不自律。 | 改为环境、触发器、反馈回路和最小实验。 |
| F13 | 只摘要不转化 | 复述材料，没有形成原则、动作或判断。 | 提炼可迁移原则和应用边界。 |
| F14 | 自信编造 | 没有来源却写成确定事实。 | 标注未核查或执行检索。 |
| F15 | 过度要求 | 低能量场景下给高负荷任务。 | 降到 2–10 分钟最小动作。 |

## 小步修改协议

每次修改 [系统元提示词](%E7%B3%BB%E7%BB%9F%E5%85%83%E6%8F%90%E7%A4%BA%E8%AF%8D%2043f03c54805d82ec9a4501cd351c1372.md) 前，先填写这 5 行：

```
失败样例：
失败类型：
要改的规则：
预期改善：
回归样例：
```

硬规则：

- 一次只修一个主要失败类型。
- 不为单个样例硬编码规则。
- 修改后至少回归 3 个旧样例。
- 如果连续 3 次修改都无法改善，先怀疑样例标准或任务定义，而不是继续加长系统提示词。
- 如果问题能通过评分表、样例或流程解决，不要继续堆提示词规则。

## 下一次使用方法

当你觉得某次 AI 回答“不对劲”时，不要直接说“优化一下”。先把它归入失败类型库，再用 16 分评分表打分；只有定位到明确失败类型后，才修改系统元提示词。
