# AI落地规范｜命名、内容、排版

<aside>
📌

本页是后续 AI 在该笔记系统内进行**新建、改名、合并、降级**时必须遵守的统一规范。

目标只有一个：**减少母页泛滥，压缩抽象词滥用，让每一页的角色一眼可判定。**

</aside>

## 一、保留词的唯一含义

### 1. OS

- 只用于**全局级 / 跨域级操作系统页面**
- 作用是统筹主线、入口、约束，不承载细碎执行细节
- **默认全局只保留 1 个活跃 OS**；若有第二个，也必须是明确的子域 OS，且需用户明确确认
- **不能**和“总纲 / 系统 / 引擎 / 架构”叠加命名

### 2. 总纲

- 只用于**某一领域的顶层方向页**
- 作用是回答：这个领域当前要解决什么、优先级是什么、入口在哪里
- 不承担完整 SOP，不堆大量机制解释
- 一个领域**最多 1 个活跃总纲**

### 3. 系统

- 只用于**可重复执行的闭环页面**
- 页面内必须至少有：目标、步骤、约束、复盘中的 3 项
- 适合训练、饮食、工作流、复盘、决策等执行页
- 如果只是理念解释或资料汇总，**不能叫系统**

### 4. 引擎

- 只用于**能稳定产出某类结果的生成模块 / 处理模块**
- 必须能回答清楚：输入是什么、处理机制是什么、输出是什么
- 引擎是“供其他页面调用的模块”，**不是总入口母页**

### 5. 架构

- 只用于**结构说明、索引说明、关系图、历史结构页**
- 架构页负责解释系统怎么组织，不负责日常执行
- 架构页默认视为**说明层 / 历史层 / 附录层**，不应占据前台高频入口

## 二、硬规则（AI 必须遵守）

1. **一个页面标题只能有 1 个核心保留词**
    - 允许：人生OS、健康总纲、饮食执行系统、内容引擎、架构说明
    - 禁止：人生OS系统、健康总纲引擎、训练系统架构、内容OS总纲
2. **没有闭环执行结构的页面，不得命名为“系统”**
3. **不是跨域总入口的页面，不得命名为“OS”**
4. **“引擎”只能用于模块页，不能用来冒充入口页**
5. **“架构”默认不是执行页，也不是日常第一入口**
6. **若页面只是旧版本、说明稿、整理稿，优先降级为“历史版本｜”或“附录｜”，不要继续抬高命名级别**
7. **AI 遇到命名冲突时，不得自行拍板创造新抽象词，必须先向用户确认**

## 三、推荐命名层级

### A. 全局层

- `人生OS v2.0`
- `P0｜执行底盘重建 30 天`

### B. 领域入口层

- `健康主线｜总入口`
- `商业主线｜总入口`
- `学习主线｜总入口`

### C. 执行层

- `饮食执行系统`
- `训练执行系统`
- `周复盘系统`
- `提示词执行协议`

### D. 研究 / 说明层

- `健康研究总览`
- `架构说明｜健康域`
- `附录｜笔记整理`
- `历史版本｜系统架构`

## 四、默认命名模板

### 1. 入口页

- 模板：`[领域]主线｜总入口`
- 用途：只放当前主线、入口链接、使用顺序

### 2. 执行页

- 模板：`[对象]执行系统`
- 用途：日常照着做的页面

### 3. 规则页

- 模板：`[对象]执行协议` / `[对象]规则清单`
- 用途：短而硬的动作规则，不扩写成母页

### 4. 研究页

- 模板：`[对象]研究总览` / `[对象]机制参考`
- 用途：原理、机制、证据、延伸阅读

### 5. 历史页

- 模板：`历史版本｜[原名]`
- 用途：保留来源，不再承担主入口职责

### 6. 附录页

- 模板：`附录｜[主题]`
- 用途：收纳整理稿、补充材料、低频说明

## 五、内容风格规则

### 1. 页面角色要在开头说清

- 正文开头必须用 1 个 callout 说明：**本页是什么 / 不是什么 / 应该何时打开**
- 禁止一上来就进入大量抽象论述

### 2. 先结论，后解释

- 默认顺序：**定位 → 规则 → 执行 → 例外 → 附录**
- 不允许先写一大段背景，再让用户自己猜怎么用

### 3. 每一页只服务一个主要任务

- 入口页不兼任研究页
- 执行页不兼任历史资料库
- 研究页不兼任总入口

### 4. 语言要短、硬、可执行

- 优先写动作句，不写空泛口号
- 每个 bullet 尽量只表达 1 个动作 / 1 个判断
- 避免连续大段排比式鸡汤

## 六、排版规则

### 1. 标题层级

- 页面正文**不重复 H1**
- 主结构只用 `##`
- 子结构用 `###`
- 一页建议控制在 **3–6 个主区块**

### 2. 段落长度

- 连续正文尽量不超过 **4 行**
- 能拆成列表就不写成长段
- 能拆成表格就不堆横向比较段落

### 3. Callout 颜色语义固定

- 蓝色：定位 / 总结
- 黄色：边界 / 警告 / 降级说明
- 绿色：执行 / 清单 / DoD
- 红色：红线 / 禁止项

### 4. 加粗规则

- 只加粗：阈值、动作、角色、结论
- 不整段加粗，不滥用强调

### 5. Emoji 规则

- 每页最多 1–2 个主要 emoji
- emoji 只辅助识别，不替代命名

## 七、AI 执行协议

### AI 在新建或改名页面前，必须先做 4 步判断

1. 这页是**入口页 / 执行页 / 研究页 / 历史页 / 附录页**中的哪一种？
2. 这页是否真的配得上使用 `OS / 总纲 / 系统 / 引擎 / 架构` 中的某一个词？
3. 当前同域是否已经有一个同角色页面？如果有，优先合并或降级，而不是并列新建
4. 若涉及结构冲突、角色冲突、命名升级，**先问用户，不得自作主张**

## 八、禁止事项

- 禁止新建标题含两个以上保留词的母页
- 禁止把“整理稿 / 收集稿 / 说明稿”命名成系统级页面
- 禁止为了显得高级而滥用 OS / 引擎 / 架构
- 禁止在未确认冲突时擅自做并列总入口
- 禁止把历史页伪装成当前主线页

## 九、一句话判定法

- **能统筹全局，才叫 OS**
- **能定义方向，才叫总纲**
- **能反复执行，才叫系统**
- **能稳定产出，才叫引擎**
- **只解释结构，就叫架构**
