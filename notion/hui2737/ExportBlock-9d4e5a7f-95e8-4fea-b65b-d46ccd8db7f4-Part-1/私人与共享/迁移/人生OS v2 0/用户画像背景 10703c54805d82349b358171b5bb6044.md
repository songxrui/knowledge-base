- 已验证经验：`D:\AI` 可以作为总目录，但最好不要直接作为 uv 项目；真实项目应放在独立子目录，如 `D:\AI\ai-coding-lab`，并在子目录内创建自己的 `pyproject.toml`。如果子目录没有 `pyproject.toml`，uv 可能向上读取父目录 `D:\AI\pyproject.toml`，导致 `requires-python = ">=3.14"` 继续影响当前项目。

# uv 环境隔离踩坑与解决（2026-05-07 追加经验）

- **父目录 pyproject.toml 会反向污染**：D:AIpyproject.toml 存在时，子项目（如 D:AILark-Formatter-mainLark-Formatter-main）若没有自己的 pyproject.toml，uv 会向上读取父目录配置，导致 requires-python 被错误约束、依赖装到错误环境。这是 D:AI 作为总目录时的高频踩坑
- **隔离解决方法**：在子项目根写最小 pyproject.toml 切断向上查找。模板：`[project]` name + version + `requires-python=">=3.12,<3.13"` + `dependencies=[]`；`[tool.uv]` `package=false`。同时写 `.python-version=3.12` 锁版本
- **项目只有 requirements.txt 时的标准修复路径**：① 写最小 pyproject.toml 隔离父目录 ② `Remove-Item .venv -Recurse -Force -ErrorAction SilentlyContinue` ③ `uv venv --python 3.12` ④ `uv pip install -r requirements.txt` ⑤ 验证三连：`uv run python --version` 显示 3.12.x + 三大核心包 import 通过 + 项目自身核心模块 import 成功
- **D: 根有 Lib/Scripts/pyvenv.cfg 是误建的孤儿 venv**（曾经在 D: 根目录跑过 `uv venv` 或 `python -m venv`），不影响项目但会让在 D: 目录下的 uv run 命中错位 venv，可选清理：`Remove-Item D:\Lib, D:\Scripts, D:\.gitignore, D:\CACHEDIR.TAG, D:\pyvenv.cfg -Recurse -Force`
- **新 PowerShell 必须重设环境变量**：`$env:PROJECT_ROOT` 等不会跨会话保留，每次开新窗口都要重新 `Set`，否则 `Set-Location $env:PROJECT_ROOT` 会因空值报错
- **PowerShell 写文件首选 Out-File -Encoding utf8 -NoNewline**：避免 BOM 和多余换行污染配置文件
