# DBS 工程系统深层拆解 — 从Skill到产品的完整工程模式

> 研究日期: 2026-06-11
> 监察: compile-and-verify + skill-review

## 一、四层工程骨架

### 第1层: 规则层 (scaffold/rules/ — 6个文件)
定义了系统的"宪法"：
- 内容单元字段规范: 13个通用字段 + 每种类型专属字段
- 关系规则: 仅4种关系（回应/解释/证明/冲突），各有方向约束
- 去重与版本规则: 规范重复检测和版本管理
- 处理流程: 10步标准顺序
- 新增文稿进入系统流程: 新内容如何入库
- 来源命名规范: 统一引用格式

**工程启示**: 规则先行。db不是先写Skill再补规则，而是在Skill内部固化了规则层。

### 第2层: 模板层 (templates/ — 7个YAML模板)
每种内容单元有精确的YAML frontmatter模板：
- 通用字段: id/type/title/source_documents/source_authors/themes/keywords/status/canonical/version/created_at/updated_at/relationships
- 类型专属: QST有question_text/question_type; CON有concept_definition; OPI有core_claim; CAS有case_subject; SOL有target_problem/action_steps

**工程启示**: 模板不是"建议格式"，是"强制规范"。这是db系统能被Agent自动处理的基础。

### 第3层: 工具层 (tools/ — 10个Node.js脚本)
每个脚本职责单一：
| 脚本 | 职责 | 输入 | 输出 |
|------|------|------|------|
| init-content-system.js | 初始化工程骨架 | 目标目录 | 完整目录+scaffold |
| generate-source-registry.js | 注册源文件 | 素材目录 | 来源注册表CSV |
| rebuild-processing-ledger.js | 重建处理台账 | 已处理CSV | 待处理清单 |
| extract-sample-units.js | 抽取内容单元 | 源文件路径 | 5类内容单元+主题地图+装配稿 |
| generate-unit-draft.js | 生成单个单元草稿 | 类型+日期+标题 | YAML模板填充 |
| assemble-topic-from-units.js | 装配主题 | 单元ID列表 | 主题装配稿 |
| generate-link-map.js | 生成关系索引 | 内容单元库 | 关系CSV+总览 |
| generate-duplicate-candidates.js | 去重候选 | 内容单元库 | 去重CSV+审计 |
| fill-obsidian-links.js | Obsidian直链 | 内容单元库 | 更新后的文件 |
| summarize-system.js | 系统摘要 | 项目根目录 | JSON统计 |

**工程启示**: 每个脚本独立可运行，不依赖其他脚本。这使系统可以分步调试。

### 第4层: 构建层 (tools/build-skills.sh)
自动化打包流水线：
```
21个Skill目录 → 按使用场景分组 → 每个Skill打包为.zip → 合集打包 → 生成README
```
特殊设计:
- 自动解析SKILL.md中的参考文档引用并打包
- 按场景分目录（必装入口/看商业问题/做内容/进阶-*）
- 输出路径可自定义

## 二、Skill编排模式: 路由器+专长Skill

### dbs（路由器）的设计哲学
- 不做诊断、不做分析、不给建议 — 只做路由
- 15条路由规则，每条绑定明确的用户意图信号
- 边界清晰: 需求不在路由表→直接告知能力范围
- 失败处理: 3种失败模式 + 对应兜底动作

### Skill间关系拓扑
```
dbs (路由器)
├── 商业诊断组
│   ├── dbs-diagnosis (诊断)
│   ├── dbs-benchmark (对标)
│   ├── dbs-deconstruct (拆概念)
│   ├── dbs-goal (审目标)
│   ├── dbs-good-question (好问题)
│   ├── dbs-slowisfast (慢方法)
│   └── dbs-action (执行力)
├── 内容创作组
│   ├── dbs-content (内容诊断)
│   ├── dbs-hook (开头优化)
│   ├── dbs-xhs-title (小红书标题)
│   └── dbs-ai-check (AI味检测)
├── 进阶组
│   ├── dbs-content-system (内容工程)
│   ├── dbs-chatroom (聊天室)
│   ├── dbs-decision (决策系统)
│   ├── dbs-learning (交互学习)
│   └── dbs-agent-migration (Agent迁移)
└── 状态管理组
    ├── dbs-save (存档)
    ├── dbs-restore (恢复)
    └── dbs-report (报告)
```

## 三、G1-G6 收费产品门禁系统

db在每个收费Skill末尾嵌入6道门禁：

| 门禁 | 检查项 | 含义 |
|------|--------|------|
| G1 大小 | body ≥800 bytes | 不低于最小有效负载 |
| G2 触发层 | 含触发词/不适用场景/边界 | 能被正确触发 |
| G3 可执行 | 核心流程可执行 | 不是空话 |
| G4 验证 | 有验证清单 | 结果可检查 |
| G5 失败兜底 | ≥3失败模式 | 出错了有退路 |
| G6 安全 | 无硬编码凭据 | 不会泄露秘密 |

这说明db不是"写了个提示词就发布"，而是把Skill当作**收费软件产品**在管理。

## 四、可提取的通用工程模式

### 模式1: 自包含模式
每个Skill = SKILL.md + 可选templates/ + 可选scaffold/ + 可选tools/ + 可选docs/
→ 单个Skill可独立运行，不依赖共享目录

### 模式2: 渐进复杂度模式
- 简单Skill: 仅SKILL.md (~3-8KB)
- 中等Skill: SKILL.md + templates/ (~8-15KB)
- 重型Skill: SKILL.md + scaffold/ + templates/ + tools/ + docs/ (~15-35KB)

### 模式3: 路由器模式
- 一个薄路由器（不做实际工作，只做分发）
- 多个专长Skill（每个只做一件事）
- 路由器维护路由表，专长Skill维护专业知识

### 模式4: 门禁模式
- 每个产出版本必须通过G1-G6
- 门禁不是"做完再检查"，是"嵌入Skill内部的自检"

### 模式5: 构建自动化模式
- 源码目录(SKILL.md+依赖) → build脚本 → 可分发的.zip
- 版本号追踪（VERSION文件）
- 按场景分组打包

## 五、对董辉Skill工程的启示

### 当前董辉Skill状态（对标分析）
| 维度 | db Skill | 董辉 Skill |
|------|---------|-----------|
| 数量 | 21个 | ~150个（含大量导入） |
| 自包含 | ✅ 每个独立可运行 | ⚠️ 部分依赖外部 |
| 路由器 | ✅ dbs统一入口 | ❌ 无统一路由 |
| 门禁系统 | ✅ G1-G6 | ⚠️ skill-review存在但不严格执行 |
| 构建自动化 | ✅ build-skills.sh | ❌ 无 |
| 分组组织 | ✅ 6个使用场景分组 | ❌ 混乱 |
| 版本管理 | ✅ VERSION+SKILL.md版本号 | ❌ 无 |

### 优先改进项
1. **建路由器**: 为董辉的150个Skill建统一路由入口
2. **执行门禁**: 每个收费级Skill必须过G1-G6
3. **分组重组**: 按使用场景分组（内容创作/商业诊断/工程基建/学习研究）
4. **去重裁军**: 150→保留核心50个，其余归档或合并
5. **版本化**: 每个Skill加版本号和更新日志
