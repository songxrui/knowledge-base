# dbs-deconstruct 概念拆解 | "AI Skill 产品"

> 使用skill: dbs-deconstruct (维特根斯坦语言哲学+奥派经济学方法论)
> 拆解对象: "AI Skill 产品" 这个概念
> 时间: 2026-06-08

---

## 语言的错位

"AI Skill产品" 这个词组有一个内在张力：
- "Skill" → 技能，能力的单元，像"会骑自行车"
- "产品" → 可交付物，像"一辆自行车"

把这两个词拼在一起，就像说"骑自行车能力产品"。

问题不在skill够不够好，在于我们用错了语言游戏。

---

## 把概念拆成5个不可再分的变量

### 变量1: 知识载体
- **本质**: 一段文本指令，告诉模型"怎么做某件事"
- **可指物**: SKILL.md 文件
- **边界**: 不是代码(代码是执行，skill是指引)，不是数据(数据是被处理对象)
- **类比**: 菜谱 vs 厨师 vs 食材。Skill是菜谱。模型是厨师。数据是食材。

### 变量2: 使用场景
- **个人用**: 创建者=使用者，不需要"产品化"
- **团队用**: N个使用者，需要"标准化"
- **公开用**: 陌生人用，需要"信任基础设施"
- **边界**: 同一份SKILL.md在不同场景下，对"产品化"的要求完全不同

### 变量3: 信任需求
- **0信任**: 自己写的自己用，错了自己修 → 草稿够用
- **低信任**: 团队用，口头沟通能补缺 → Alpha级够用
- **高信任**: 陌生人在不同环境用，不能沟通 → GA级必需

### 变量4: 维护成本
- **存量成本**: 7种文件一旦建立，每次改SKILL.md都要同步改7个地方
- **增量收益**: examples让新人更快上手，tests让升级更安全
- **边界**: 7种文件的ROI取决于skill的使用频率和变更频率。一个每天用、每周改的skill，7种文件的ROI是负的。

### 变量5: 完成定义
- **功能完成**: SKILL.md写完 → 能干活了
- **信任完成**: 7种文件齐全 → 别人敢用了
- **传播完成**: GitHub + 文档 + 案例 → 别人能找到并选择了
- **三个完成定义对应三个不同的"产品"概念**

---

## 推论

1. **"产品"是一个光谱，不是开关。** 不存在"某个skill突然从草稿变成产品"的时刻。
2. **7种文件是信任基础设施，不是功能。** 就像银行存款准备金——它不是银行赚钱的业务，但没它储户不敢存钱。
3. **"99%不配叫产品"这个论断的问题在于：它用最高标准(公开交付)衡量了所有skill，忽略了"个人用够了"的场景。**
4. **更好的问法不是"这个是产品吗？"而是"这个skill的当前完成度匹配它的使用场景吗？"**
