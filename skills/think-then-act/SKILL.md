# think-then-act — 先想后动

> 核心戒律：意图不明 = 立即中断，向用户索取必要信息。禁止猜测、禁止假设、禁止"先做着再说"。

## 一句话定义

在执行任何行动之前，强制完成意图理解→方案规划→计划审核→监工执行四个阶段。不理解的意图 = 不执行。

---

## 阶段 0：意图准入闸（HARD GATE）

不通过禁止进入阶段 1。

闸门判据：
- 用户的真实目标是什么？（不是字面任务，是最终想要的结果）
- 用户明确说了做什么和不做什么吗？
- 存在歧义吗？（一个词可以有多种理解？一个指令可以有多种执行方式？）
- 缺失关键信息吗？（路径？格式？范围？约束？）
- 用户的假设和我的假设一致吗？

任一判据不满足 → 立即中断，向用户提问。最多 3 个问题，按优先级排序。

提问模板：
⚠️ 意图澄清 — 需要确认以下信息后再执行：
1. [最关键的不明确点]
2. [次关键的不明确点]
3. [如果还有]

禁止行为：
- 禁止说"我理解为…"然后继续（理解 ≠ 确认）
- 禁止在用户未回复前自行推断
- 禁止用封闭式问题引导用户
- 禁止跳过此闸直接规划

---

## 阶段 1：方案规划（Think）

输入：阶段 0 确认后的清晰意图。输出：结构化行动方案。

### 1.1 目标还原

表层任务：[用户字面说了什么]
深层目标：[用户真正想达成什么结果]
成功标准 DoD：[什么状态算"做完了"]
失败模式：[哪些结果绝对不能出现]

### 1.2 约束清单

硬约束（不可违反）：[路径限制/格式要求/兼容性]
软约束（尽量满足）：[性能偏好/风格偏好]
环境假设（标注"待确认"）：[假设了什么工具/环境/权限可用]

### 1.3 分步行动计划

Step 1: [动作]
  工具：[具体工具/skill名称]
  输入：[需要什么前置信息/文件]
  输出：[这一步产出什么可验证的结果]
  验证：[如何确认这一步做对了]
  失败兜底：[如果这步失败，替代方案是什么]

Step N: 同上格式

### 1.4 边界声明

本次明确做：[纳入范围]
本次明确不做（即使相关）：[排除范围 + 不做理由]

---

## 阶段 2：计划审核（Review）

闸门判据（全部满足才进入阶段 3）：
- 每一步有明确的工具/skill 调用描述
- 每一步有可验证的输出定义
- 每一步有失败兜底方案
- 范围边界已明确标注
- 无计划外的隐性步骤
- 计划顺序无循环依赖

审核不通过 → 回到阶段 1 修正，最多 2 轮。2 轮后仍不通过 → 输出当前方案草案 + 未解决的问题，请求用户决策。

---

## 阶段 3：监工执行（Act）

执行纪律：
1. 严格按照阶段 1 的计划顺序执行，禁止跳步
2. 每完成一步，确认该步输出通过验证，再进入下一步
3. 若执行中发现假设错误 → 暂停，回到阶段 1 更新计划
4. 禁止在计划外引入额外步骤（即使"顺手优化一下"也不行）
5. 每步完成后更新进度

进度追踪格式：
| Step | 状态 | 验证结果 | 备注 |
|------|------|---------|------|
| 1/N  | ✅   | [验证通过] |      |
| 2/N  | 🔄   | 执行中... |      |
| 3/N  | ⏳   |          |      |

---

## 阶段 4：终验（Verify）

终验清单：
- 所有计划步骤已执行完毕
- 每步验证通过（无跳过的失败）
- 最终产出与阶段 1 的 DoD 一致
- 无计划外的额外改动
- 无遗漏的边界外操作

终验不通过 → 定位失败步骤 → 从该步骤重新执行（最多 1 次全流程重试）→ 仍不通过 → 输出部分完成状态 + 失败说明，请求用户决策。

---

## 失败处方表

| 失败场景 | 处理动作 |
|---------|---------|
| 阶段 0 意图不清 | 中断，最多 3 个问题提问用户 |
| 用户回复仍不明确 | 再问 1 轮（最多 2 轮澄清），2 轮后标注假设继续 |
| 阶段 1 规划不出可行方案 | 标注阻塞点，询问用户是否有替代路径 |
| 阶段 2 审核 2 轮不通过 | 输出方案草案 + 未解决问题，请求用户决策 |
| 阶段 3 执行中发现假设错误 | 暂停，回退阶段 1 更新计划 |
| 阶段 3 某步失败 | 按该步失败兜底执行；兜底也失败则记录并继续（除非前置依赖则中断） |
| 阶段 4 终验不通过 | 定位失败步骤，从该步重新执行（最多 1 次），仍不通过则输出部分完成状态 |
| 用户中途改变意图 | 接受新意图，从阶段 0 重新开始（不沿用旧计划） |

## 与其他 Skill 的联动

think-then-act（意图→规划→监工）
    ↓
compile-and-verify（验证执行正确性）
    ↓
content-guard（≥2 文件修改时的防退化门禁）

边界：
- 不做执行后验证 → 由 compile-and-verify 负责
- 不做逐文件质量扫描 → 由 content-guard 负责
- 不做任务优先级排序 → 由 agent-sort 负责

## 调用示例

### 示例 1：意图清晰，直接规划

用户：帮我把 D:\KnowledgeBase 下所有 .md 文件中的"赋能"替换为"增强"

阶段 0：意图清晰（目标/路径/操作明确）→ 通过
阶段 1：Step 1: rg 扫描命中文件 → Step 2: 逐个替换 → Step 3: Git 提交
阶段 2-4：按计划执行并终验

### 示例 2：意图模糊，中断提问

用户：优化一下我的内容

阶段 0：
- 真实目标？→ 不清楚（"内容"指什么？）
- 存在歧义？→ 是（"优化"指什么？）

→ 中断提问：
⚠️ 意图澄清：
1. "内容"具体指什么？（文章/社交媒体帖子/知识库条目？路径？）
2. "优化"的目标是什么？（去AI味/增强传播力/补充证据？）
