# 慢方法 — Skill建设: 为什么"更慢"才是"更快"

> 产出方法: dbs-slowisfast (慢方法诊断)  
> 核心逻辑: 找到看起来更慢但长期更快的路径

---

## 快方法 vs 慢方法: Skill建设版

### 快方法 (大家都在做的)
- 看到别人skill好用 → 复制粘贴SKILL.md → 立即调用
- 出问题了 → 抱怨"这个skill不行" → 继续找下一个
- 3个月后 → 50个skill, 不知道哪个有用哪个没用

**结果**: 看起来快(10秒装一个)，实际慢(3个月后全是技术债务)

### 慢方法 (看起来费时, 长期更快)
- 选一个核心skill → 花30分钟补齐触发词+git+tests
- 跑5个真实任务验证效果
- 确认有用 → 归档到核心链
- 确认没用 → 删除, 不留技术债务
- 每月审计一次, 清理过时skill

**结果**: 看起来慢(30分钟一个)，实际快(每个skill都是验证过的资产)

---

## 为什么"慢方法"长期更快 (3个证据)

### 证据1: OpenSkillEval — 坏skill拖累性能
OpenSkillEval证明: 最差skill比不用skill差0.28分。
→ 快方法: 装了50个skill, 可能有10个在拖累你
→ 慢方法: 只留20个验证过的skill, 每个都在帮你

### 证据2: SkillsBench — 精选优于自生成
精选skill +16.2pp / 自生成skill -1.3pp。
→ 快方法: 每天自生成2-3个skill (质量不可控)
→ 慢方法: 每周打磨1-2个skill (质量可控)

### 证据3: 上下文经济学
100个Draft skill = 5000 tokens metadata (每次对话)。
→ 快方法: 5000 tokens被75%的概率不会被调用的skill占据
→ 慢方法: 2000 tokens被100%验证过的高质量skill占据

---

## 慢方法的5步操作

### Step 1: 不装新skill, 先删坏的 (30分钟)
- 列出所有skill
- 标记: 30天未调用 → 候选删除
- 标记: 有负效果 → 立即删除
- 目标: 从147个 → 50个

### Step 2: 给核心skill打GA门禁 (每skill 30分钟)
- 选最常用的10个skill
- 逐条过7文件标准
- 补齐缺失项
- 目标: 10个GA级skill

### Step 3: 建编排层 (2小时)
- 为10个核心skill建orchestrator
- Orchestrator负责路由, 用户不直接面对10个skill
- 目标: 1个orchestrator + 10个核心skill = 11个激活项

### Step 4: 建调用日志 (1小时)
- 记录每次skill加载时的: token消耗/任务结果/有用否
- 每周review一次日志
- 目标: 数据驱动的skill生命周期管理

### Step 5: 每月审计 (30分钟/月)
- 跑skill-overseer审计
- 清理过期skill
- 归档不常用的(不删, 移到.archived/)

## 时间算账

| 方法 | 初始时间 | 每次会话损失 | 3个月总成本 |
|------|:---:|------|------|
| 快方法(147个skill) | 0 | 5000 tokens × 每天10次 = 50K tokens/天 | 3个月≈450万tokens + 坏skill干扰 |
| 慢方法(50个核心) | 10小时 | 2000 tokens × 每天10次 = 20K tokens/天 | 3个月≈180万tokens + 零干扰 |
| **节省** | -10小时 | | **270万tokens + 质量增益** |

10小时的"慢投入"换来3个月270万tokens的节省 + 系统性质量提升。这就是"慢即快"的核心逻辑。
