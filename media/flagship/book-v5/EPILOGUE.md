# 后记：这本书是怎么写出来的——以及你如何自己造一本

> 本书是"10h著书引擎"的产物——一个用git commit时间戳作为唯一工时证据、用weread+exa作为唯一信源、用"立反合落地"作为唯一写作方法的著书系统。

## 方法论：为什么"用AI写书"不是让AI替你写

大多数人理解的"AI写书"是"给一个prompt，让AI生成8章"——这产生的是无法验证的空话合集，引用的书可能不存在，页码是编造的，数据没有出处。

这本书的写作流程反过来：

1. **先锁定信源，再展开写作**：每章先列"该主题人类最顶尖的5-8本书"，通过weread API实调确认书架上有，或通过exa实搜确认书籍存在+内容可查。不是"AI觉得应该引用什么"，而是"真实存在的书说了什么"。

2. **AI只做编排和检索，不动笔写核心论点**：AI的角色是——搜索信源、提取关键观点、对比不同书的说法、把零散的笔记组织成结构。但每个"应该怎么做"的最终判断来自原著，不是来自模型。

3. **每一条都经过反驳复审**：写完"立"（核心主张）之后，强制走"反"——这个方法在什么情况下失效？最强的反例是什么？边界在哪？——这一步消灭了大部分"听起来对但经不起推敲"的内容。

4. **git时间戳是唯一的工时证明**：不看AI自述"我工作了多久"，只看git log的首尾commit间隔。如果所有commits挤在40分钟内——那就是40分钟的产品，不是10小时的产品。

## 如果你也想造一本自己的书

你不需要成为"作家"——你需要的是：

1. **一个明确的主题**：你真正理解、有真实经验的领域（不是"我觉得应该写什么"）
2. **5-8本你能验证的名著**：weread书架、实体书、公开论文——只要能查证
3. **一个git仓库**：每写完一节就commit，用时间戳作为你诚实工作的证据
4. **一套验证方法**：每写完一个观点就问"这个观点来自哪本书？""失效条件是什么？""删掉后读者少了什么？"

本书的著书引擎（prompt+信源流程+质量控制）已在GitHub开源：songxrui/knowledge-base。你可以用它启动你自己的著书项目。
