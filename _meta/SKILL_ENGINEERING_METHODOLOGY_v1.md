## 一、Skill 的第一性理解

### 你原来的理解对在哪、缺在哪

你说"skill = 加了工具且有摘要的提示词"——**对了一半**。对在：body 确实是提示词性质的程序性指令。缺在两个关键维度：

| 你的理解 | 实际 |
|---------|------|
| 提示词 | ✅ body 是指令，但不是"一次塞满上下文"的提示词——是**按需载入的程序性记忆** |
| 加了工具 | ⚠️ 工具只是资源层的一种；更核心的区分在触发层 |
| 有摘要 | ⚠️ description 不是"摘要"——是**模型的召回与路由信号**，决定 skill 是否被激活 |

### 三层结构（替代"prompt+工具+摘要"）

```
① 触发层 (Trigger Layer)
   ├─ name: 唯一标识符
   ├─ description: 召回信号——何时该触发、何时不该触发
   └─ 权重: 0 token（始终在上下文，但不占正文）

② 程序层 (Program Layer)  
   ├─ SKILL.md body: 操作手册——被触发后才载入上下文
   └─ 权重: 轻量时 ~500 token，重型时 ≤2000 token

③ 资源层 (Resource Layer)
   ├─ scripts/: 可执行脚本
   ├─ references/: 参考文档（按需读取，不预载）
   └─ templates/: 可复制模板
```

### 关键事实（SkillOpt 核心洞察）

📄 **宿主模型参数冻结，Skill 文档是唯一可变外部状态。** 模型权重不能改，但 skill 文本可以。所以 skill 的质量 = 把多少"正确的程序性知识"压进最小 token，并让它在对的时刻被召回。

🧠 **推论**：skill 工程本质是"用文本编辑做模型微调"——你在不动模型权重的情况下，通过改变外部程序性记忆来改变模型行为。这解释了为什么"多改少说"的 skill 有效，而"堆砌概念"的 skill 无效——后者没有给模型提供可执行的程序。

### 一句话定义

> **Skill = 对的时间点被载入上下文的、可验证的最小可执行程序文本。**
> 三个要素缺一不可：对的触发时机 × 可执行规则 × 可验证标准。

## 二、触发条件设计规范

### 把"精准触发"拆成两个独立优化目标

| 目标 | 定义 | 度量 |
|------|------|------|
| **召回 (Recall)** | 该触发时触发了 | 漏触发次数 / 应触发总次数 |
| **精度 (Precision)** | 不该触发时不触发 | 误触发次数 / 总触发次数 |

两者通常互斥——扩大触发词提升召回但降低精度。策略：**先保精度，再提召回**（误触发的代价 > 漏触发，因为误触发会污染上下文、浪费 token）。

### Description / 触发条件写作模板

```yaml
description: |
  [一句话说清这个 skill 做什么，15-25 字]
  
  适用场景:
  - [场景1：具体的用户行为/任务描述]
  - [场景2]
  
  不适用场景:
  - [明确排除的场景1——避免语义重叠误触发]
  - [明确排除的场景2]
  
  触发词: [关键词1, 关键词2, 关键词3, 关键词4, 关键词5]
  
  边界: 与 [相邻skill名] 的区别是 [一句话边界]。
       与 [另一个相邻skill] 的区别是 [一句话边界]。
  
  正例:
  - "[用户会说的话]" → 触发
  - "[用户会说的话]" → 触发
  
  反例:
  - "[用户会说的话——但不应触发]" → 不触发，转到 [正确的skill]
  - "[用户会说的话——但不应触发]" → 不触发
```

### 触发词设计规则

- **≥5 个明确触发词**，含中文和英文变体
- 触发词必须是**用户会自然使用的词**，不是你的内部术语
- 避免单个泛化词（如"优化"）——加限定词（如"skill 优化"、"提示词优化"）
- 与相邻 skill 的触发词要有**互斥检查**：取两个 skill 的触发词集合，交集须为空

### 误触发 / 漏触发诊断与修法

| 症状 | 诊断方法 | 修法 |
|------|---------|------|
| **误触发**（不该触发时触发了） | 查看上下文日志，找到触发时用户实际说了什么 | ① 在"不适用场景"中加入该情形；② 缩减触发词范围；③ 在 description 中加边界语句 |
| **漏触发**（该触发时没触发） | 用户明确提到某 skill 或描述其功能，但 skill 未被加载 | ① 增加同义触发词；② 降低触发门槛（放宽 wording）；③ 检查 name 是否与触发词匹配 |

## 三、熵增膨胀治理

### 膨胀三来源

| 来源 | 表现 | 根因 |
|------|------|------|
| **规则冗余** | 同一条规则用不同说法写了 3 遍 | 迭代时只加不改，旧规则未删除 |
| **语义重叠** | 两个 skill 的 body 有 40%+ 内容相同 | 新建 skill 时复制了旧 skill 的通用规则，未做去重 |
| **抽象空话** | "你应该仔细检查""要认真思考" | 写规则时依赖自然语言直觉，未落到可验证动作 |

📄 SkillOpt 数据：优化后的 skill 中位大小约 900 token（约 700 中文字），上限建议 2000 token。超过这个数时，每增加一条规则，边际收益递减——因为模型注意力被稀释。

### "一条规则能留下"的硬标准（4 条全过 = 留，任一不过 = 删或改）

- [ ] **可操作**：读完这条规则，模型知道具体做什么动作（含命令/参数/判断条件），不是"应该注意XX"
- [ ] **可验证**：有一个布尔判定来判断"这条规则被遵守了还是违反了"
- [ ] **不重复**：用 `rg` 搜索整个 skill 库，这条规则的实质内容没有在其他规则/skill 中出现
- [ ] **删了会掉分**：在留出任务上做 A/B 对比——有这条规则 vs 没有这条规则——删了分数下降 >0 才保留（🧠 推断：对开放任务用固定 rubric 人评替代自动评分）

### Token 预算检查表（每新增/修改 skill 时执行）

- [ ] body 是否 ≤ 2000 token（约 1500 中文字）？
- [ ] 每 200 token 是否至少含 1 个可执行命令/参数/判断条件？
- [ ] 有没有任何"你可以试试""建议考虑""可能有用"类模糊词？（→ 改为具体动作或删除）
- [ ] 有没有短词分行、空行制造篇幅？（≥3 个连续短行 → 合并）
- [ ] description 是否 ≤ 80 字？

### 多 Skill 去重与边界划分方法

1. **内容哈希去重**：取所有 skill body，对每段规则做语义哈希（提取关键动词+宾语），标记相似度 >70% 的规则对——人工判定是合并还是各保留
2. **边界矩阵**：建一张 N×N 表格，每个 cell 填两个 skill 的关系：
   - `A 包含 B` → 合并 B 到 A 或删 B
   - `A 概率触发 B` → 在 A 的"不适用场景"中加触发词互斥
   - `独立` → 保留
3. **层级化**：如果 3+ 个 skill 共享同一组通用规则，提取为 `_shared/SHARED_RULES.md`，各 skill 通过引用（`参见 SHARED_RULES.md §3`）而非复制来使用——但注意引用路径必须绝对，确保 AI 能解析

## 四、手动版 SkillOpt 迭代循环

### 核心映射（神经网络训练 → 文本编辑）

| 神经网络概念 | SkillOpt 文本编辑等价 | 你的手动操作 |
|-------------|---------------------|------------|
| 前向传播 (forward) | rollout | 拿当前 skill 跑 N 个真实任务，记录每次成功/失败 |
| 梯度 (gradient) | reflection | 分析失败模式，写出 ≤3 条改进方向（不改文档） |
| 权重更新 (update) | edit | 对 SKILL.md 做 add/delete/replace，**每轮 ≤4 条** |
| 学习率 (lr) | textual LR | 📄 默认 4 条/轮，cosine decay（后期衰减到 1-2 条/轮） |
| 验证集 (validation) | validation gating | 在留出任务上跑，**分数没涨 = 回滚本轮改动** |
| 早停 (early stop) | — | 连续 2 轮不涨 = 停止，不再加规则 |
| Checkpoint | git commit | 每轮 commit，保留回滚点 |

### 可执行 SOP（5 阶段循环）

```
┌──────────────────────────────────────────────────────┐
│  阶段1: ROLLOUT（不碰文档，只跑任务）                    │
│  用当前 skill 跑 ≥5 个真实任务，记录：                   │
│  - 任务描述 + 预期行为 + 实际行为 + 成功/失败            │
│  - 失败时标注：漏触发？误触发？规则不清晰？规则缺失？       │
├──────────────────────────────────────────────────────┤
│  阶段2: REFLECTION（不碰文档，只写分析）                  │
│  对失败案例：                                          │
│  - 单条失败 = 偶发 → 不处理                              │
│  - 同一模式 ≥2 次 = 系统性缺陷 → 提取改进方向             │
│  输出 ≤3 条改进方向（每条：问题 + 建议 + 优先级）          │
├──────────────────────────────────────────────────────┤
│  阶段3: EDIT（改文档，每轮 ≤4 条）                       │
│  只做 3 种编辑：                                       │
│  - ADD: 增加新规则（因漏触发或规则缺失）                  │
│  - DELETE: 删除无效规则（因误触发或冗余）                 │
│  - REPLACE: 重写模糊规则为具体规则                       │
│  📄 纪律：超过 4 条 → 选最重要的 4 条，其余入 buffer     │
├──────────────────────────────────────────────────────┤
│  阶段4: VALIDATION（不改文档，只跑验证）                  │
│  在留出任务（≥3 个，至少含 1 个已知失败案例）上跑：        │
│  - 失败案例全部修复 → 本轮通过                           │
│  - 新增失败 → 本轮作废，回滚                             │
│  - 无变化 → 本轮作废，回滚                               │
│  通过 → git commit；不通过 → git reset --hard           │
├──────────────────────────────────────────────────────┤
│  阶段5: REJECTED BUFFER + META UPDATE                  │
│  - 记录本轮被否的改动到 rejected.md（避免下轮重犯）       │
│  - 每 3-5 轮做一次 meta update：跨版本纵向对比            │
│    · 哪些规则反复被改？→ 可能根本规则有问题，重写而非修   │
│    · 哪些规则从未被触发？→ 死规则，删除                   │
│    · Token 是否在涨？→ 如每轮都只加不删，强制执行减法轮   │
└──────────────────────────────────────────────────────┘
```

### 开放任务下的代理判据（替代自动评分）

📄 SkillOpt 明确声明：自动验证依赖 exact match 或自动评分器——**开放性任务不适用**。

🧠 以下为迁移到开放任务的替代方案（推断，非论文实证）：

| 方案 | 操作 | 适用 |
|------|------|------|
| **固定 Rubric 人评** | 为 skill 定义 3-5 个评分维度（如触发精准度/规则可执行性/输出格式符合度），每维度 0-3 分，rollout 后人工打分。≥80% 满分 = 通过 | 任何开放任务 |
| **A/B 双版本盲评** | 拿旧 skill(A) 和新 skill(B) 各跑同一任务，隐藏版本号，人工选"哪个输出更好"或"平局"。B 赢 >50% 才接受 | 高频使用 skill |
| **关键失败用例回归集** | 从历史 rollout 中收集 10-20 个明确失败案例，每次验证必须全部通过——任一不通过 = 回滚 | 成熟 skill 的维护 |

### 克制是第一纪律

📄 SkillOpt 核心发现：每轮改动 ≤4 条 + cosine decay 是收敛的关键。改动越多 ≠ 越优——未收敛的改动互相冲突，导致 skill 质量震荡。

- 每轮 ≤4 条改动（默认）
- 前 3 轮可以 4 条，之后衰减到 2 条，最后衰减到 1 条
- 连续 2 轮验证不涨 → 停止，不收工继续加（早停信号）
- 被否改动记入 buffer，避免下轮重犯
- **先手动跑通 3 轮完整循环，再考虑自动化**

## 五、改造他人导入 Skill 的流程清单

按序执行，每步可验证，不跳步：

```
Step 1: 读懂原意图
  ├─ 这个 skill 解决什么问题？
  ├─ 它的目标模型是谁（Claude/Codex/Cursor）？和你的 DeepSeek v4 Pro 有何差异？
  └─ 输出：1 句话的"核心意图"——如果说不清 = skill 本身有问题

Step 2: 剥离空话与冗余
  ├─ 用第 3 节的硬标准逐条过规则
  ├─ 标记所有"可以试试/建议考虑/可能有用/认真思考/仔细检查"→ 改写或删除
  └─ 输出：纯规则清单（不含任何空话）

Step 3: 按第二部分模板重写触发条件
  ├─ 用你的模型和目标场景重写 description
  ├─ 添加"不适用场景"和边界语句（防误触发）
  ├─ 检查与已有 skill 的触发词是否有交集——有交集 = 加边界区分
  └─ 输出：新 frontmatter

Step 4: 压缩到 token 预算
  ├─ 正文 ≤2000 token
  ├─ 合并含义相同的规则
  ├─ 非核心规则 → 移到 references/（按需读取）
  └─ 输出：压缩后的 SKILL.md

Step 5: 在真实任务上跑 validation
  ├─ ≥5 个真实任务
  ├─ 记录：触发是否正确？输出是否可验证？是否有规则被忽略？
  └─ 输出：validation 记录

Step 6: 与已有 skill 去重并划清边界
  ├─ 跑内容哈希去重检查
  ├─ 有重叠 → 在双方的 description 中明确边界
  └─ 输出：更新后的边界矩阵

Step 7: 标注版本与改动理由
  ├─ 在 SKILL.md 末尾加版本块：版本号 + 日期 + 改动理由 + 来源
  └─ 输出：最终 SKILL.md
```

### 导入决策门槛（任一不过 = 不导入）

- [ ] 原始 skill 的核心意图能在 1 句话说清
- [ ] 至少有 2 条可操作、可验证的规则
- [ ] 与现有 skill 库无 >50% 语义重叠
- [ ] 你近期（2 周内）有 ≥3 次真实使用场景
