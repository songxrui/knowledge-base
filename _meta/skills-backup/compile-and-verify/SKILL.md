# compile-and-verify R3 — 任务编译+交付验证

> 完整文档: `SKILL.md.bak` (4.7KB)

## 核心原则

把模糊任务编译为可量化目标变量，执行后对照验收，确保交付物准确完整无幻觉。

---

## 3 Phase 流程

### Phase 1: 编译 (入口关)
1. **质疑需求**: 用户要求中哪些是伪需求/过度格式/错误假设？
2. **目标量化**: 每个目标写成 "X 达到 Y" 格式 (如 "测试通过率 ≥95%")
3. **复杂度分流**:
   - **simple**: 单步/无歧义/纯信息 → 跳过 Phase 2 直接执行
   - **complex**: 多步/有歧义/涉及文件修改 → 进入 Phase 2

### Phase 2: 执行 (按编译后的量化目标执行)

### Phase 3: 验证 (出口关)

| # | 检查项 | 验证方式 |
|---|--------|---------|
| 1 | 目标达成 | 逐项对比量化目标 vs 实际产出 |
| 2 | 信息完整 | 关键字段/步骤无遗漏 |
| 3 | 无幻觉 | 数据/引用可回溯到来源 |
| 4 | 格式正确 | 输出格式符合规范 |
| 5 | 无破坏 | 未影响不相关的文件/功能 |

### 验证报告格式
```
## 验证报告
### 目标达成
- [目标1] ✅/❌ (实际值/偏差)
- [目标2] ✅/❌ (实际值/偏差)

### 完整性检查
- 关键字段覆盖率: X/Y

### 最终判定
✅ 通过 / ❌ 未通过(原因:XX)
```

## 失败模式与兜底

| 失败模式 | 原因 | 兜底 |
|---------|------|------|
| 量化目标过于模糊 | 用户原始需求不清晰 | 回 Phase 1.2 用 "X达到Y" 格式重写 |
| 验证项遗漏关键检查 | 检查清单不完整 | 补充 domain-specific 检查项 |
| 简单任务被过度编译 | 复杂度判断失误 | 回 Phase 1.3 重新分流 |

## 约束
- 不负责内容质量(dbs-content)或代码安全(security-review)
- 编译阶段不执行实际修改，只输出编译计划

## G1-G6

| 门禁 | 状态 |
|------|------|

| G2 触发层 | ✅ |

| G4 验证(验证报告) | ✅ |
| G5 失败兜底(3模式) | ✅ |
| G6 安全 | ✅ |
