## 三大病灶

### 病灶1 · D1触发精准（P0·阻断收费发布）

**现象**: frontmatter description有触发词但无"不适用场景"和"与相邻skill边界"

**证据**:
```
当前 description:
"批改/修复/优化任何文本文件前必须激活。三条铁律：双向防偏、最小手术、回退测试。
触发词：修改文章、修复内容、优化文案、批量编辑..."
→ 缺少：不适用场景 / 反例 / 与compile-and-verify的边界
```

**修复**（1行，D1: 6→8，总评→84 A级，G2达标）:
在description末尾追加不适用场景+边界

### 病灶2 · D6信息密度（P1·体验优化）

**现象**: 342行中90行<20字（26.3%），多为单列表头行和分隔线

**证据**:
```
|---|--------|---------|  (分隔线，信息量为0)
| C1 | ...    | ...     |  (表头行，结构必要但信息密度低)
```

**修复**: 合并短行表格为内联描述；压缩G1-G7门禁中的重复"关联缺陷"

### 病灶3 · D7边界清晰（P1·防误触发）

**现象**: 缺少与compile-and-verify/content-truth-lock的显式分工边界

**证据**:
```

但 frontmatter 的 description/触发条件中未声明与这些相邻skill的触发优先级
→ 可能导致两个skill同时被触发，上下文冗余
```
